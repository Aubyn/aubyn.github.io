k3s 上手 ​

以下是我为你设计的一套由浅入深的问题引导体系，共 10 个问题，分为 4 个阶段。每个问题先提出，再给出解析回答，帮助你从 Docker 基础平滑过渡到 k3s 的容器编排实战。

第一阶段：建立认知（Docker → k3s 的桥梁） ​

问题 1：我已经会用 Docker 跑前端镜像了，k3s 和 Docker 是什么关系？它是替代 Docker 的吗？ ​

解析回答：

不是替代关系，而是互补和分层的关系。

  • Docker 是容器运行时 + 镜像构建工具。你写的 Dockerfile、构建的镜像、docker run 启动容器，这些在 k3s 世界里依然完全适用。
  • k3s 是轻量级 Kubernetes 发行版，核心能力是容器编排——即管理多个容器的部署、扩缩容、服务发现、负载均衡、故障自愈等。

你可以这样理解：

  • Docker 解决的是"怎么把应用装进盒子并运行"
  • k3s 解决的是"怎么管理成千上万个盒子，让它们协同工作"

k3s 内部实际上使用 containerd（Docker 底层同款容器运行时）来启动容器，所以你的 Docker 镜像在 k3s 中可以直接使用，无需修改。

问题 2：既然 k3s 是 Kubernetes 的轻量版，它"轻"在哪里？为什么适合我这种本地/个人使用场景？ ​

解析回答：

k3s 的"轻"体现在几个关键设计：

方面传统 K8sk3s
二进制大小多个组件，体积大单二进制文件 ~50MB，一个进程内封装所有控制平面组件
存储后端必须 etcd（需额外维护）默认 SQLite3，也支持 etcd/MySQL/Postgres
依赖需要大量系统组件和配置内置 Flannel（网络）、CoreDNS（DNS）、Traefik（Ingress）、Helm 控制器等
资源需求至少 2CPU/2GB 起步512MB 内存即可运行，树莓派都能跑
安装复杂，多步骤一条命令安装：curl -sfL https://get.k3s.io | sh -

对于你（Arch Linux 单机、前端镜像部署），k3s 的优势是：

  • 一台电脑就能跑完整的 Kubernetes 集群
  • 学习成本比传统 K8s 低得多
  • 为以后上云（阿里云 ACK、腾讯云 TKE 等）打下完全兼容的基础

第二阶段：动手安装（Arch Linux 环境） ​

问题 3：在 Arch Linux 上安装 k3s，有什么需要特别注意的地方？和 Ubuntu 有什么不同？ ​

解析回答：

Arch 是滚动发行版，内核较新，有几个注意点：

1. cgroup 版本 Arch 默认使用 cgroup v2（检查：stat -fc %T /sys/fs/cgroup/ 应输出 cgroup2fs）。k3s v1.21+ 已原生支持，一般无需额外配置。若遇到老版本问题，可在 GRUB 参数中添加 systemd.unified_cgroup_hierarchy=1。

2. 关闭 swap Kubernetes 要求关闭 swap：

bash
sudo swapoff -a
# 注释掉 /etc/fstab 中的 swap 行

3. iptables/nftables 兼容性 Arch 默认用 nftables，k3s 可能需要 iptables 兼容层：

bash
sudo pacman -S --needed iptables-nft

4. 安装方式（二选一）

方式 A：AUR（推荐，方便更新）

bash
yay -S k3s-bin
sudo systemctl enable --now k3s

方式 B：官方脚本

bash
curl -sfL https://get.k3s.io | sudo sh -

5. 配置 kubectl 访问

bash
mkdir -p ~/.kube
sudo cp /etc/rancher/k3s/k3s.yaml ~/.kube/config
sudo chown $(id -u):$(id -g) ~/.kube/config
kubectl get nodes  # 验证

问题 4：安装完成后，我该怎么验证 k3s 真的跑起来了？核心组件都在做什么？ ​

解析回答：

执行以下命令验证：

bash
# 1. 查看节点状态
kubectl get nodes -o wide
# 应显示你的 Arch 机器为 Ready 状态

# 2. 查看系统组件（k3s 内置的核心服务）
kubectl get pods -n kube-system

你会看到类似这些 Pod：

  • coredns-*：集群 DNS 服务，让 Pod 之间可以通过服务名互相访问
  • local-path-provisioner-*：本地存储供应器，提供持久化卷
  • metrics-server-*：资源监控，支持 kubectl top 命令
  • traefik-*：默认的 Ingress 控制器（HTTP/HTTPS 路由）

跑一个测试验证：

bash
kubectl run test --image=nginx --restart=Never
kubectl get pod test -w  # 等待状态变为 Running
kubectl delete pod test

这些组件都是 k3s **内置"电池"**的一部分，传统 K8s 需要你一个个手动安装。

第三阶段：核心概念（从 Docker 思维转换到 k3s 思维） ​

问题 5：在 Docker 里我用 docker run -p 8080:80 暴露端口，在 k3s 里怎么让前端服务能被外部访问？ ​

解析回答：

k3s 里不直接暴露容器端口，而是通过多层抽象：

外部请求 → Ingress → Service → Pod(容器)

核心概念对比：

Dockerk3s/Kubernetes
docker runPod（最小调度单元，可包含多个容器）
-p 8080:80 端口映射Service（ClusterIP/NodePort/LoadBalancer）
nginx 反向代理Ingress（HTTP 路由规则）
docker-compose.ymlDeployment（声明式部署配置）

实战示例：部署一个前端应用并暴露访问

  1. 创建 frontend-deployment.yaml：
yaml
apiVersion: apps/v1
kind: Deployment
metadata:
    name: frontend
spec:
    replicas: 2 # 运行2个副本，实现高可用
    selector:
        matchLabels:
            app: frontend
    template:
        metadata:
            labels:
                app: frontend
        spec:
            containers:
                - name: frontend
                  image: your-frontend-image:latest
                  ports:
                      - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
    name: frontend-service
spec:
    selector:
        app: frontend
    ports:
        - port: 80
          targetPort: 80
    type: ClusterIP # 集群内部访问
---
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
    name: frontend-ingress
spec:
    rules:
        - host: frontend.local
          http:
              paths:
                  - path: /
                    pathType: Prefix
                    backend:
                        service:
                            name: frontend-service
                            port:
                                number: 80
  1. 应用配置：
bash
kubectl apply -f frontend-deployment.yaml
  1. 本地测试（修改 /etc/hosts 添加 frontend.local 127.0.0.1）：
bash
curl http://frontend.local
# 或者直接用 NodePort 方式快速测试

关键点：k3s 默认安装了 Traefik 作为 Ingress Controller，所以 Ingress 规则可以直接生效。

问题 6：Docker 镜像怎么放到 k3s 里用？需要推送到 Docker Hub 吗？本地镜像行不行？ ​

解析回答：

有三种方式，按场景选择：

方式 1：使用远程镜像仓库（推荐用于生产/团队协作）

yaml
containers:
    - name: frontend
      image: docker.io/yourname/frontend:v1.0

k3s 节点会自动从 Docker Hub 拉取。

方式 2：本地构建 + 导入 k3s（适合单机开发测试）

bash
# 在 Arch 上构建镜像
docker build -t my-frontend:latest .

# 导出并导入到 k3s 的 containerd
docker save my-frontend:latest > my-frontend.tar
sudo k3s ctr images import my-frontend.tar

# 然后在 yaml 中使用
image: my-frontend:latest
imagePullPolicy: Never  # 或 IfNotPresent，避免去远程拉取

方式 3：使用本地镜像仓库（如 Harbor，适合小团队） 在 k3s 中配置 registries.yaml 指向私有仓库。

对于你的场景（本地前端部署），方式 2 最方便；如果后续要上 CI/CD，再过渡到方式 1 或 3。

问题 7：k3s 的 Deployment、Service、Ingress 这些 YAML 配置太复杂了，有没有更简单的方式管理前端应用？ ​

解析回答：

有，推荐渐进式学习路径：

阶段 1：kubectl 命令式（快速验证）

bash
kubectl create deployment frontend --image=my-frontend --replicas=2
kubectl expose deployment frontend --port=80 --type=NodePort

阶段 2：YAML 声明式（理解原理，必学） 上面的 Deployment + Service + Ingress 组合，这是 K8s 的核心思想，建议掌握。

阶段 3：Helm 包管理（复用配置） k3s 内置 Helm 控制器。Helm 就像 k3s 世界的"apt/yay"，可以一键安装复杂应用：

bash
# 例如安装一个前端应用模板
helm repo add bitnami https://charts.bitnami.com/bitnami
helm install my-nginx bitnami/nginx

阶段 4：Kustomize / 可视化工具

  • Kustomize：k3s 内置，用于管理多环境（dev/staging/prod）配置覆盖
  • Kuboard / Lens：图形化管理界面，降低 YAML 编写负担

建议：先硬写 YAML 理解原理，再用 Helm 提升效率。

第四阶段：进阶实战（运维、CI/CD、排障） ​

问题 8：我的前端应用更新了，k3s 怎么实现"不停机更新"？回滚怎么做？ ​

解析回答：

这是 k3s/K8s 相比 Docker 的巨大优势——声明式滚动更新。

更新镜像版本：

bash
# 方式 1：直接修改 deployment
kubectl set image deployment/frontend frontend=my-frontend:v2.0

# 方式 2：修改 YAML 后重新 apply
kubectl apply -f frontend-deployment.yaml

k3s 会自动执行滚动更新：

  1. 创建新版本的 Pod
  2. 健康检查通过后，逐步替换旧 Pod
  3. 旧 Pod 优雅终止，服务不中断

查看更新进度：

bash
kubectl rollout status deployment/frontend
kubectl get pods -w  # 实时观察新旧 Pod 交替

回滚到上一个版本：

bash
kubectl rollout history deployment/frontend  # 查看版本历史
kubectl rollout undo deployment/frontend    # 回滚到上一版
kubectl rollout undo deployment/frontend --to-revision=2  # 回滚到指定版本

前端场景特别有用：发版后发现 bug，秒级回滚，用户无感知。

问题 9：我想实现"代码提交 → 自动构建镜像 → 自动部署到 k3s"，该怎么搭建这个流水线？ ​

解析回答：

这就是 CI/CD + GitOps 的核心场景。一个最小可行方案：

架构：

GitHub 代码仓库 → GitHub Actions → 构建 Docker 镜像 → 推送到镜像仓库 → 更新 k3s Deployment

GitHub Actions 示例 (.github/workflows/deploy.yml)：

yaml
name: Build and Deploy
on:
    push:
        branches: [main]
jobs:
    build:
        runs-on: ubuntu-latest
        steps:
            - uses: actions/checkout@v4

            - name: Build and push image
              run: |
                  docker build -t ghcr.io/yourname/frontend:${{ github.sha }} .
                  echo ${{ secrets.GITHUB_TOKEN }} | docker login ghcr.io -u ${{ github.actor }} --password-stdin
                  docker push ghcr.io/yourname/frontend:${{ github.sha }}

            - name: Update k3s deployment
              run: |
                  # 使用 kubeconfig 连接到你的 Arch k3s
                  mkdir -p ~/.kube
                  echo "${{ secrets.KUBECONFIG }}" > ~/.kube/config
                  kubectl set image deployment/frontend frontend=ghcr.io/yourname/frontend:${{ github.sha }}
                  kubectl rollout status deployment/frontend

Arch 机器上的准备：

  1. 将 /etc/rancher/k3s/k3s.yaml 内容复制为 GitHub Secret KUBECONFIG
  2. 修改 k3s.yaml 中的 server: https://127.0.0.1:6443 为你 Arch 机器的内网/公网 IP
  3. 确保防火墙开放 6443 端口（或仅允许 GitHub Actions IP）

更高级的 GitOps 方案：使用 ArgoCD（可部署在 k3s 内），实现"仓库即唯一真相源"，自动同步集群状态。

问题 10：k3s 运行出问题了，我该怎么排查？有没有常用的"救命命令"？ ​

解析回答：

以下是前端开发者必备的 k3s 排障工具箱：

1. 查看 Pod 状态和事件（最常用）

bash
kubectl get pods -A                    # 所有命名空间的 Pod
kubectl describe pod <pod-name>        # 详细事件日志，看为什么起不来
kubectl logs <pod-name> --tail=100     # 查看应用日志
kubectl logs <pod-name> -c <container> # 多容器 Pod 指定容器
kubectl logs <pod-name> --previous     # 崩溃前日志

2. 查看 k3s 系统服务

bash
sudo systemctl status k3s              # 服务状态
sudo journalctl -u k3s -f              # 实时日志

3. 网络排障

bash
kubectl get svc -A                     # 查看服务暴露的端口
kubectl get ingress -A                   # 查看路由规则
kubectl port-forward svc/frontend 8080:80  # 本地快速测试服务

4. 资源使用

bash
kubectl top node                         # 节点资源（需 metrics-server，k3s 已内置）
kubectl top pod

5. Arch 特有排障

bash
# containerd 起不来
sudo modprobe overlay
echo "overlay" | sudo tee /etc/modules-load.d/overlay.conf

# 网络策略不生效
sudo modprobe nf_conntrack nft_compat xt_conntrack

6. 完全重置（最后手段）

bash
sudo /usr/local/bin/k3s-uninstall.sh   # 卸载
# 然后重新安装
