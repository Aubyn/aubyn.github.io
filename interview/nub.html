Nub ​

根据您提供的官方 FAQ 资料，我为您梳理了一条由浅入深的学习路径，通过 8 个关键问题，带您逐步理解 Nub 的本质、能力、原理和生态定位。

第一阶段：基础认知（它是什么？） ​

Q1：Nub 到底是什么？它是一个新的 JavaScript 运行时（如 Node/Bun/Deno）吗？

答： 绝对不是。Nub 是一个基于 Rust 编写的命令行工具（CLI），它的核心定位是 “运行在 Node 之上的现代化工具链层”。

  • 本质：它是一个“全能工具包”，用于替代你日常使用的 node（运行文件）、npm run（运行脚本）和 npx（运行本地 CLI）。
  • 运行时真相：当你执行 nub index.ts 时，底层实际调用的执行引擎仍然是你电脑上原本安装的那个 node 二进制文件。Nub 没有自己的 JS 引擎，没有修改 Node 源码，也不是一个 fork。
  • 结论：Nub 不是替代 Node，而是“增强型”的 Node。它只是在你和 Node 之间加了一层智能的指挥调度。

第二阶段：功能体验（用它到底有什么好处？） ​

Q2：如果底层还是 Node，那我换成 nub 命令后，能立刻获得哪些看得见摸得着的好处？

答： 最大的好处是开箱即用的现代开发体验，无需再手动搭建繁琐的工具链。具体体现在：

  1. 原生 TypeScript/JSX 支持：直接运行 .ts、.tsx 文件，并且完整支持 TypeScript 语法（枚举、命名空间、装饰器等），而不仅仅是去掉类型。它会自动读取你的 tsconfig.json 中的 paths 和 baseUrl 映射。
  2. 自动加载环境变量：自动按优先级（如 .env.local > .env）加载环境变量，不再需要 dotenv-cli。
  3. 多格式导入：可以直接 import 导入 YAML、TOML、JSON5 等格式文件。
  4. 自动注入 Polyfill：自动填补 Node 缺失的 Web API，如 Temporal、URLPattern、浏览器标准的 Worker 等。
  5. 极快的启动速度：因为本体是 Rust 编译的二进制文件，执行 nub run 比 pnpm run 冷启动快 24 倍，执行 nubx 比 npx 快 19 倍。

第三阶段：实战替代（它能替换我现有工具链中的哪些东西？） ​

Q3：既然它号称“全能工具包”，那我现有的 tsx、nodemon、npx 是不是都可以扔掉了？

答： 是的，资料明确指出 Nub 可以直接替换以下两大类工具：

  • 第一类（直接替换）：tsx / ts-node（用 nub script.ts）、nodemon（用 nub watch script.ts）、npx（用 nubx）、dotenv-cli（自动加载）、以及 nvm/fnm 等 Node 版本管理工具（Nub 能根据 .node-version 自动调配）。
  • 第二类（借力 Node 现代版本）：因为 Nub 强制要求较新的 Node 版本（18.19+），所以你可以免费享受到 Node 内置的 fetch、WebSocket、fs.glob、crypto.randomUUID 等特性，从而移除 node-fetch、ws、glob 等老旧 npm 包。

第四阶段：核心机制（它为什么能做到不破坏兼容性？） ​

Q4：Nub 在底层到底施展了什么魔法，使得它既能增强功能，又能保证“代码在 Node 上能用，在 Nub 上就一定能用”？

答： 它的魔法在于完全依赖 Node 官方公开的扩展接口，而不是黑科技 hack。工作流程如下：

  1. Rust 指挥：Rust CLI 读取你的 tsconfig.json 和 .env。
  2. 标准衍生（Spawning）：它找到你 PATH 里的 node 二进制，并通过 --import 参数注入一个 Nub 的预加载脚本。
  3. 官方钩子（Hooks）：预加载脚本调用 Node 官方的 module.registerHooks() （Node 22.15+ 稳定）来注册一个加载钩子。
  4. 转译与执行：当你 import 文件时，这个钩子调用 N-API 插件（基于 oxc 转译器）实时转译 TS/JSX，然后把结果交给 V8 执行。
  5. 结论：这一整套流程全部使用 module.registerHooks、--import、NODE_OPTIONS、N-API 等公开的、稳定的 Node 原生接口。因此，它不是在一个沙盒里模拟 Node，而是实实在在地在 Node 进程里运行你的代码。

第五阶段：包管理与 Monorepo（它如何管理依赖和大型项目？） ​

Q5：Nub 有自己的包管理器吗？它怎么处理 pnpm 的 workspace（Monorepo）？

答： Nub 自带包管理引擎（nub install），但它的策略极其温和：

  • 锁文件兼容：它直接读取并写入你项目现有的锁文件（pnpm-lock.yaml、package-lock.json、甚至 bun.lock）。这意味着你和团队可以混用 pnpm install 和 nub install，不会产生冲突，无需迁移。
  • Monorepo 一等公民：nub run 和 nubx 完美支持 -r（递归）和 --filter 语法。最惊艳的是性能：在 30 个包的 Monorepo 中，pnpm -r run build 要为每个包支付约 150ms 的启动开销；而 nub run -r build 只支付一次 Rust 启动开销，0 次 Node/Pnpm 的重复启动开销。

第六阶段：对比竞品（面对 Bun 和 Deno，Nub 的独特价值在哪？） ​

Q6：现在有 Bun 和 Deno 这样的现代一体化运行时，Nub 跟它们相比，核心优势是什么？

答： 核心优势在于 “不加锁”的增量式升级。

  • Bun/Deno 是从零重写的运行时（Bun 用 JSC，Deno 用 V8）。它们有自己独特的 API（如 Bun.serve、Deno.env）。一旦你的代码用了这些 API，就与 Node 生态产生了割裂。
  • Nub 坚守 Node 地盘。你的代码里绝对不会出现 globalThis.nub 或 nub:* 这样的特殊 API。Nub 提供的所有增强（TS、Env）都可以通过手动配置在原生 Node 上复现。如果你把 nub 从项目中移除，代码本身无需任何改动，回到原生 Node 照样跑。它提供的是“消失后不留痕迹”的兼容性。

第七阶段：风险与兜底（如果不想用了或者出 Bug 了怎么办？） ​

Q7：如果 Nub 有 Bug，或者我决定回归原生命令行，代价大吗？

答： 代价几乎为零，这是 Nub 设计的核心原则。

  • 无锁定（No Lock-in）：Nub 不往你的 package.json 或 tsconfig.json 里添加任何额外字段，也不产生任何专属的锁文件。
  • 兼容模式：Nub 贴心地提供了 --node 标志（如 nub run --node test）。这个标志会剥离所有运行时增强（不转译 TS、不加载 Env），只保留 Nub 的任务编排能力（如 workspace 调度）去调用原生 Node，方便你排查问题。
  • 回退路径：如果 Nub 明天消失，你最多是把 TypeScript 编译步骤加回 CI，把 dotenv 加回代码，而你的业务代码本身一个字都不用改。

第八阶段：安装与日常使用（最后，我该如何开始？） ​

Q8：简单总结一下，我日常怎么安装和使用它？

答： 非常简单：

  • 安装：npm install -g --ignore-scripts=false @nubjs/nub（或 brew install nubjs/tap/nub）。
  • 日常替换：
    • 跑 TS/JS 文件：以前 node index.js → 现在 nub index.ts。
    • 跑 package.json 脚本：以前 npm run dev / pnpm run dev → 现在 nub run dev。
    • 跑本地 bin：以前 npx prisma generate → 现在 nubx prisma generate。
  • 版本要求：增强模式需要 Node 18.19+；如果是老项目，可以用 nub run --node 降级为纯编排模式。
