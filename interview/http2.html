HTTP/2 ​

这是一份针对前端面试中 HTTP/2 的深度学习指南。我将通过 7 个关键问题，按照 “问题背景 -> 核心机制 -> 性能优化 -> 实践应用” 的逻辑层层递进，帮助你建立完整的知识体系。

阶段一：起源与动机 ​

Q1: 为什么我们需要 HTTP/2？HTTP/1.1 有什么致命缺陷？ ​

解析： 这个问题采用 黄金思维圈 的逻辑顺序回答：先看 Why（痛点），再看 What（现状）。 1. 队头阻塞： 在 HTTP/1.1 中，虽然支持持久连接，但在同一个 TCP 连接中，请求必须串行发送。前一个请求未响应，后一个请求只能等待。如果一个请求耗时过长，后续所有请求都会被阻塞。 2. 头部庞大： HTTP/1.1 的头部通常是纯文本，且包含大量重复信息（如 Cookie、User-Agent）。每次请求都必须携带完整的头部，导致带宽浪费。 3. 连接数限制： 浏览器为了解决队头阻塞，会对同一域名建立多个 TCP 连接（通常 Chrome 限制 6 个）。但这带来了额外的 TCP 握手开销和连接竞争。 结论： HTTP/2 的诞生就是为了解决 HTTP/1.1 的传输效率低和资源开销大的问题。

阶段二：核心机制 ​

Q2: HTTP/2 是如何实现“多路复用”的？ ​

解析： 这是 HTTP/2 最核心的特性。回答逻辑采用 程序运行视角的执行先后顺序。 1. 基本概念：

  • 帧：HTTP/2 通信的最小单位。每个帧包含帧头，标识该帧属于哪个流。
  • 流：已建立的连接内的双向字节流，承载一条或多条消息。
  • 消息：对应 HTTP/1.1 中的请求或响应，由一个或多个帧组成。 2. 工作原理： 在 HTTP/1.1 中，数据是按“消息”发送的（必须发完整个请求）。 在 HTTP/2 中，数据被拆解为“帧”。
  • 步骤一：客户端将请求 A 和请求 B 分别拆解为帧（A1, A2... B1, B2...）。
  • 步骤二：这些帧可以在同一个 TCP 连接上交错发送。
  • 步骤三：服务端接收后，根据帧头中的 Stream ID 重新组装成完整的请求 A 和请求 B。 效果： 解决了队头阻塞，使得一个连接可以同时处理无数个请求。

Q3: 既然 HTTP/2 多路复用这么强，为什么还需要“头部压缩”？ ​

解析： 这个问题采用 数据传输视角 分析。 虽然多路复用解决了body的传输效率，但HTTP头部依然是明文传输且重复度高。HTTP/2 引入了 HPACK 算法：

  1. 静态字典：将常用的头部名称和值组合（如 method: GET）编码为静态索引。传输时只传索引号，不传字符串。
  2. 动态字典：在连接期间，客户端和服务端共同维护一个动态表。第一次传 user-agent: Chrome 后，后续传输只需传索引号。
  3. Huffman 编码：对头部数据进行压缩。 结果： 头部体积可减少 85% 以上，极大降低了传输延迟。

阶段三：性能与实践 ​

Q4: HTTP/2 的“服务器推送”是如何工作的？前端代码如何配合？ ​

解析： 这个问题采用 用户视角交互先后顺序 来解析。 场景： 用户请求 index.html，该页面依赖 style.css。 传统模式 (HTTP/1.1)：

  1. 浏览器请求 index.html。
  2. 浏览器解析 HTML，发现需要 style.css。
  3. 浏览器发起对 style.css 的请求。
  4. 服务器返回 style.css。 (RTT 往返延迟)服务器推送模式 (HTTP/2)：
  5. 浏览器请求 index.html。
  6. 服务器预测客户端需要 style.css，在发送 HTML 的同时，主动将 style.css “推”送给浏览器。
  7. 浏览器解析 HTML 发现需要 style.css，发现本地已有缓存，直接使用。 代码示例（Node.js 服务端关键代码）：
javascript
const http2 = require("http2");
const fs = require("fs");
const server = http2.createSecureServer({
  key: fs.readFileSync("server.key"),
  cert: fs.readFileSync("server.crt"),
});
server.on("stream", (stream, headers) => {
  const path = headers[":path"];
  if (path === "/") {
    // 1. 发送 index.html
    stream.respondWithFile("index.html");

    // 2. 主动推送 style.css (关键代码)
    stream.pushStream({ ":path": "/style.css" }, (pushStream) => {
      pushStream.respondWithFile("style.css");
    });
  }
});
server.listen(8443);

Q5: 既然 HTTP/2 解决了队头阻塞，为什么还会出现 HTTP/3？ ​

解析： 这是一个深度的架构视角问题。HTTP/2 只解决了 HTTP 层的队头阻塞，但未解决 TCP 层的队头阻塞。 TCP 的局限性： TCP 是可靠传输协议，保证数据包的顺序。如果 TCP 层发生丢包，后续的数据包即使已经到达接收端，也会被内核缓存区阻塞，等待丢包重传成功后才能交付给应用层。 HTTP/3 (QUIC) 的解决方案： HTTP/3 底层不再使用 TCP，而是使用 QUIC (基于 UDP)。

  • QUIC 在传输层原生支持多流。
  • 如果流 A 丢包，只会阻塞流 A 的重组，流 B 和流 C 可以继续传输。

阶段四：开发策略 ​

Q6: 在 HTTP/2 时代，前端的“性能优化策略”发生了哪些根本性变化？ ​

解析： 回答逻辑采用 对比分析 法。

优化策略HTTP/1.1 时代的做法HTTP/2 时代的做法原因
域名分片将资源分散到多个域名下合并域名，减少 DNS 查询和 TCP 握手HTTP/2 连接利用率高，多连接反而增加开销
文件合并将 JS/CSS 打包成一个大文件适度拆分，利用缓存合并包大，修改一行全量失效；拆分后只请求变更文件，且并行下载极快
雪碧图合并小图标减少请求内联 SVG 或保留独立图片HTTP/2 请求成本低，雪碧图维护成本高
内联资源将 CSS/JS 直接写在 HTML 中避免内联，使用外部文件内联无法利用头部压缩，且无法被浏览器单独缓存

总结 ​

Q7: 如果面试官让你总结 HTTP/2 的核心优势，你会怎么说？ ​

解析： 使用 STAR 法则 组织总结：

  • Situation (背景)：HTTP/1.1 面临队头阻塞、连接数限制和头部冗余三大痛点，无法满足现代 Web 复杂应用的需求。
  • Task (任务)：提升传输效率，降低延迟。
  • Action (行动)：
    • 引入二进制分帧和多路复用，在一个 TCP 连接上并行传输。
    • 实施 HPACK 头部压缩，减少带宽消耗。
    • 支持 服务器推送，预加载资源。
    • 加密强制化（主流浏览器仅支持加密的 HTTP/2）。
  • Result (结果)：页面加载速度显著提升，网络资源利用率优化，推动了前端架构从“合并打包”向“模块化细粒度缓存”的转变。
