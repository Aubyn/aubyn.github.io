1. 2026 年前端构建有哪些变化？ ​

核心变化：Rust 化 + Vite 主导 ​

2026 年最大的变化是构建工具全面 Rust 化，以及 Vite 成为绝对主流。

具体变化点：

维度2025 及以前2026 年
主流工具Webpack 仍占大量存量Vite 6 成为新项目首选，Webpack 仅用于遗留项目
底层引擎基于 JavaScript（Webpack、Rollup JS 版）全面 Rust 化：Rolldown、Turbopack、Rspack
冷启动Webpack 中型项目 25–40 秒Vite 6 仅需 0.08 秒，HMR 延迟低至 10ms
生产构建Rollup（JS 版）Rolldown（Rust 重写 Rollup），速度提升近 10 倍
开发模式全量打包后启动基于 ESM 按需编译，直接服务源码

关键工具格局：

  • Vite 6：Rust 内核（Rolldown）+ ESM 原生 + 10ms HMR → 新项目首选
  • Turbopack：Vercel 出品，增量编译极致 → Next.js 默认
  • Rspack：字节出品，Webpack 兼容 + Rust 性能 → 存量迁移
  • Webpack 6：仅用于遗留项目维护 → 逐步淘汰

面试回答要点：

"2026 年前端构建最大的变化是工具链全面 Rust 化。Vite 6 通过 Rolldown 统一了开发和生产构建，冷启动从秒级降到毫秒级。Webpack 虽然生态成熟，但新项目几乎都会选择 Vite，Rspack 则成为存量 Webpack 项目迁移的最佳方案。"

问题 1：Vite 为什么比 Webpack 快？ ​

核心差异：开发模式的根本不同 ​

维度WebpackVite
启动方式全量打包后启动按需编译，直接服务源码
模块处理构建完整的依赖图利用浏览器原生 ESM
HMR 机制基于模块依赖图重新编译精准替换单个模块
冷启动时间25–40 秒（中型项目）300–800 毫秒

详细原理 ​

Webpack 的慢：

  1. 全量打包：启动时扫描所有入口文件，递归构建完整的模块依赖图
  2. 递归编译：每个文件经过 Loader 转换 → Plugin 处理 → 合并 Chunk
  3. HMR 传播：修改一个文件，需要重新编译整个依赖链

Vite 的快：

  1. 按需编译：浏览器请求哪个模块，Vite 才编译哪个模块
  2. 原生 ESM：利用浏览器 import 能力，无需打包成 bundle
  3. HMR 精准替换：修改文件后，仅重新编译该模块，通过 WebSocket 推送更新
  4. 预构建依赖：首次启动时用 esbuild 预编译第三方库，缓存到 node_modules/.vite

代码示例 ​

javascript
// Webpack: 启动时全量编译
// 修改 src/components/Button.jsx
// → 重新编译 Button → 重新编译引用 Button 的页面 → 重新编译整个 chunk

// Vite: 精准替换
// 修改 src/components/Button.jsx
// → 仅编译 Button.jsx → 通过 WebSocket 推送更新 → 浏览器热替换

面试回答要点 ​

"Vite 比 Webpack 快的原因在于开发模式的根本差异。Webpack 启动时需要全量打包构建依赖图，而 Vite 利用浏览器原生 ESM 按需编译——浏览器请求哪个模块就编译哪个。HMR 时 Webpack 需要重新编译整个依赖链，Vite 只需精准替换单个模块。另外 Vite 用 esbuild（Go 编写）预编译依赖，比 Babel 快 10–100 倍。"

问题 2：Tree Shaking 原理是什么？ ​

核心机制：静态分析 + 标记清除 ​

Tree Shaking 的本质是基于 ES Module 的静态结构分析，在编译时确定未使用代码并剔除。

为什么 ES Module 能做到？ ​

特性CommonJSES Module
导入时机运行时动态 require()编译时静态 import
结构分析无法静态确定依赖可以静态分析依赖图
Tree Shaking不支持支持

实现步骤 ​

1. 构建模块依赖图（Module Graph）
2. 标记所有被引用的导出（Mark）
3. 遍历所有模块，标记被使用的代码
4. 清除未标记的代码（Sweep）
5. 生成最终 bundle

sideEffects 配置 ​

json
// package.json
{
  "sideEffects": false           // 所有文件都可以安全 Tree Shaking
  // 或
  "sideEffects": ["*.css", "*.scss"]  // 这些文件有副作用，保留
}

副作用（Side Effects）：指模块执行时除了导出值外，还修改了全局状态（如 polyfill、CSS 注入）。

面试陷阱 ​

javascript
// ❌ 无法 Tree Shaking（动态导入）
const utils = require('./utils');

// ❌ 无法 Tree Shaking（动态属性访问）
import * as utils from './utils';
const method = utils[methodName];

// ✅ 可以 Tree Shaking（静态导入）
import { add, subtract } from './utils';

面试回答要点 ​

"Tree Shaking 的原理是基于 ES Module 的静态结构特性。ESM 的 import/export 是编译时确定的，构建工具可以静态分析出完整的依赖图，然后标记被引用的导出，最后清除未标记的代码。package.json 中的 sideEffects 字段告诉构建工具哪些文件有副作用需要保留。CommonJS 的 require() 是运行时动态执行的，所以无法 Tree Shaking。"

问题 3：Rollup 和 Webpack 的区别？ ​

定位差异 ​

维度RollupWebpack
设计目标打包 JavaScript 库构建复杂 Web 应用
输出格式ESM、CJS、UMD、IIFE主要是 CJS + 运行时
代码拆分支持，但较弱强大的 Code Splitting
Loader 生态简单，插件较少极其丰富
HMR不支持原生支持
Tree Shaking原生支持，效果更好支持，但配置复杂

为什么库用 Rollup，应用用 Webpack？ ​

Rollup 适合库的原因：

  1. 输出纯净：不会注入大量运行时代码
  2. Tree Shaking 友好：基于 ESM，剔除未使用代码更彻底
  3. 多格式输出：可同时输出 ESM + CJS + UMD

Webpack 适合应用的原因：

  1. Loader 丰富：CSS、图片、字体等资源都能处理
  2. Code Splitting：强大的代码分割和懒加载
  3. HMR：开发时热更新体验好
  4. 缓存策略：长效缓存、runtime 分离等

2026 年的变化 ​

Vite 开发用 Rollup（现用 Rolldown），生产也用 Rollup，打破了"库用 Rollup，应用用 Webpack"的界限。

面试回答要点 ​

"Rollup 和 Webpack 的核心区别在于设计目标：Rollup 专注打包 JavaScript 库，输出格式纯净、Tree Shaking 效果好；Webpack 专注构建复杂 Web 应用，Loader 生态丰富、支持 HMR 和强大的 Code Splitting。2026 年 Vite 用 Rolldown 统一了开发和生产构建，说明 Rollup 的能力已经足够支撑应用级构建。"

问题 4：Rspack 和 Webpack 的兼容性如何？ ​

Rspack 的定位 ​

Rspack 是字节跳动开源的 Rust 版 Webpack，目标是即插即用替换 Webpack，享受 Rust 性能提升。

兼容性对比 ​

维度兼容性说明
配置格式100%webpack.config.js 直接复用
Loader95%+大多数 Loader 无需修改
Plugin90%+主流 Plugin 兼容，部分需适配
API高compilation、compiler 等核心 API 对齐

迁移成本 ​

javascript
// webpack.config.js → rspack.config.js
// 几乎无需修改！
const path = require('path');

module.exports = {
    entry: './src/index.js',
    output: {
        path: path.resolve(__dirname, 'dist'),
        filename: '[name].[contenthash].js',
    },
    module: {
        rules: [
            {
                test: /\.js$/,
                use: 'babel-loader', // ✅ 直接可用
            },
            {
                test: /\.css$/,
                use: ['style-loader', 'css-loader'], // ✅ 直接可用
            },
        ],
    },
    plugins: [
        new HtmlWebpackPlugin(), // ✅ 兼容
    ],
};

性能对比 ​

项目WebpackRspack提升
冷启动40s5s8x
HMR3s200ms15x
生产构建60s10s6x

面试回答要点 ​

"Rspack 是 Rust 重写的 Webpack，兼容性非常高，配置文件几乎可以直接复用，Loader 和 Plugin 生态也能大部分兼容。它的优势是让存量 Webpack 项目零成本迁移，同时获得 Rust 带来的 5–10 倍性能提升。对于不想重构构建配置的老项目，Rspack 是最佳迁移方案。"

问题 5：如何优化首屏加载？ ​

优化维度：减少 + 提前 + 并行 ​

策略具体手段效果
减少资源体积Tree Shaking、Gzip、Brotli、代码分割减少传输量
减少请求数合并小文件、内联关键 CSS降低 RTT
提前加载<link rel="preload">、DNS 预解析提前建立连接
并行加载HTTP/2 多路复用、域名分片提升并发
渲染优化SSR/SSG、骨架屏、关键渲染路径提升感知速度

关键指标：Core Web Vitals ​

指标目标优化手段
LCP（最大内容绘制）< 2.5s图片优化、预加载、SSR
FID（首次输入延迟）< 100ms减少长任务、Web Worker
CLS（累积布局偏移）< 0.1图片尺寸预设、字体加载策略

实战代码 ​

html
<!-- 预加载关键资源 -->
<link rel="preload" href="/fonts/main.woff2" as="font" type="font/woff2" crossorigin />
<link rel="preload" href="/critical.css" as="style" />

<!-- DNS 预解析 -->
<link rel="dns-prefetch" href="//cdn.example.com" />
<link rel="preconnect" href="//cdn.example.com" />

<!-- 图片懒加载 -->

<!-- 关键 CSS 内联 -->
<style>
    /* 首屏需要的 CSS 直接内联在 <head> */
    .hero {
        background: url(...);
    }
</style>

面试回答要点 ​

"首屏优化从三个维度入手：减少资源体积（Tree Shaking、Gzip、代码分割）、提前加载关键资源（preload、preconnect）、优化渲染路径（SSR/SSG、关键 CSS 内联）。同时要关注 Core Web Vitals 指标，LCP 控制在 2.5s 内，FID 控制在 100ms 内，CLS 控制在 0.1 内。"

问题 6：如何分析打包体积？ ​

工具链 ​

工具适用场景特点

问题 7：长列表卡顿怎么解决？ ​

核心方案：虚拟列表（Virtual Scrolling） ​

只渲染可视区域内的元素，滚动时动态替换。

实现原理 ​

可视区域高度: 500px
每个列表项高度: 50px
可视区域内最多渲染: 500 / 50 = 10 项
实际渲染: 10 + 缓冲项(上下各 5) = 20 项

总数据: 10000 项
DOM 节点数: 20 个（而非 10000 个）

常用库 ​

库特点适用场景
react-window轻量、固定/可变高度React 项目
react-virtualized功能丰富复杂布局
@tanstack/react-virtual2026 年推荐现代 React，Headless
vue-virtual-scrollerVue 专用Vue 项目

代码示例（react-window） ​

jsx
import { FixedSizeList as List } from 'react-window';

function VirtualList({ items }) {
    const Row = ({ index, style }) => <div style={style}>{items[index].name}</div>;

    return (
        <List
            height={500} // 可视区域高度
            itemCount={items.length} // 总数据量
            itemSize={50} // 每项高度
            width="100%"
        >
            {Row}
        </List>
    );
}

进阶优化 ​

优化点方案
动态高度预估高度 + 实际测量缓存
无限滚动滚动到底部加载更多
双向滚动上下都支持无限加载
大数据量分页 + 虚拟列表结合

面试回答要点 ​

"长列表卡顿的核心原因是 DOM 节点过多导致重排重绘性能下降。解决方案是虚拟列表，只渲染可视区域内的元素加上少量缓冲项。React 中推荐 react-window 或 @tanstack/react-virtual，它们通过计算滚动位置动态渲染对应数据项，将 DOM 节点从万个降到几十个。对于动态高度场景，需要预估高度并在渲染后缓存实际高度。"

问题 8：FCP/LCP/CLS 如何优化？ ​

Core Web Vitals 详解 ​

指标含义目标优化手段
FCP（First Contentful Paint）首次内容绘制< 1.8s内联关键 CSS、减少阻塞资源
LCP（Largest Contentful Paint）最大内容绘制< 2.5s图片优化、预加载、SSR
FID（First Input Delay）首次输入延迟< 100ms减少长任务、代码分割
TTFB（Time To First Byte）首字节时间< 600msCDN、边缘计算、缓存
CLS（Cumulative Layout Shift）累积布局偏移< 0.1图片尺寸预设、字体加载策略

针对性优化 ​

LCP 优化：

html
<!-- 预加载 LCP 图片 -->
<link rel="preload" as="image" href="/hero.jpg" fetchpriority="high" />

<!-- 响应式图片 -->
<picture>
    <source srcset="hero.avif" type="image/avif" />
    <source srcset="hero.webp" type="image/webp" />
    <img src="hero.jpg" width="1200" height="600" alt="Hero" />
</picture>

CLS 优化：

html
<!-- 图片必须设置尺寸 -->
<img src="photo.jpg" width="800" height="600" alt="" />

<!-- 字体加载策略：避免 FOUT/FOIT -->
<style>
    @font-face {
        font-family: 'Custom';
        src: url('font.woff2') format('woff2');
        font-display: swap; /* 先显示后备字体，加载后替换 */
    }
</style>

FID 优化：

javascript
// 长任务拆分
function processLargeData(data) {
    const chunkSize = 1000;
    let index = 0;

    function processChunk() {
        const chunk = data.slice(index, index + chunkSize);
        // 处理 chunk
        index += chunkSize;

        if (index < data.length) {
            requestIdleCallback(processChunk); // 利用空闲时间
        }
    }

    processChunk();
}

面试回答要点 ​

"Core Web Vitals 是 Google 衡量用户体验的核心指标。FCP 优化要减少阻塞资源、内联关键 CSS；LCP 优化要预加载关键图片、使用 WebP/AVIF 格式、SSR 渲染；CLS 优化要给图片和广告预留固定尺寸、使用 font-display: swap 避免字体加载导致的布局偏移；FID 优化要拆分长任务、利用 requestIdleCallback 和 Web Worker。"

问题 10：微前端方案对比？ ​

主流方案 ​

方案原理优点缺点
qiankunJS Sandbox + HTML Entry成熟、生态丰富沙箱性能开销
Module Federation运行时模块共享共享依赖、无缝集成强耦合、构建复杂
iframe浏览器原生隔离完全隔离、简单通信困难、体验差
Web Components原生组件封装标准方案浏览器兼容性、生态弱
Micro App类 Web Component轻量、易用新兴方案

qiankun 原理 ​

1. 主应用注册子应用
2. 加载子应用的 HTML Entry
3. 解析 HTML，提取 JS/CSS
4. 创建 JS Sandbox（Proxy 代理 window）
5. 执行子应用代码
6. 路由切换时卸载子应用

Module Federation 原理 ​

javascript
// 远程应用暴露模块
// webpack.config.js (remote)
const { ModuleFederationPlugin } = require('webpack').container;

module.exports = {
    plugins: [
        new ModuleFederationPlugin({
            name: 'remoteApp',
            filename: 'remoteEntry.js',
            exposes: {
                './Button': './src/Button',
            },
            shared: ['react', 'react-dom'], // 共享依赖
        }),
    ],
};

// 主应用消费模块
// webpack.config.js (host)
module.exports = {
    plugins: [
        new ModuleFederationPlugin({
            remotes: {
                remoteApp: 'remoteApp@http://localhost:3001/remoteEntry.js',
            },
        }),
    ],
};

选型建议 ​

场景推荐方案
存量项目迁移qiankun
新项目、技术栈统一Module Federation
完全隔离、安全要求高iframe
追求标准、长期维护Web Components

面试回答要点 ​

"微前端方案主要有 qiankun、Module Federation、iframe 三种。qiankun 通过 JS Sandbox 实现子应用隔离，适合存量项目迁移；Module Federation 在运行时共享模块和依赖，适合技术栈统一的新项目；iframe 完全隔离但通信困难。2026 年趋势是 Module Federation 2.0 和原生 Web Components 的结合，减少运行时开销。"

问题 11：如何设计前端 SDK？ ​

设计原则 ​

原则说明
Tree Shaking 友好使用 ESM，避免副作用
多格式输出ESM + CJS + UMD
零依赖/少依赖减少用户安装负担
类型安全提供 TypeScript 声明
体积控制核心功能精简，扩展按需加载

目录结构 ​

my-sdk/
├── src/
│   ├── core/           # 核心功能
│   ├── utils/          # 工具函数
│   ├── plugins/        # 插件系统
│   └── index.ts        # 入口
├── dist/
│   ├── index.mjs       # ESM
│   ├── index.cjs       # CJS
│   ├── index.umd.js    # UMD
│   └── index.d.ts      # 类型声明
├── package.json
└── tsconfig.json

package.json 配置 ​

json
{
    "name": "my-sdk",
    "version": "1.0.0",
    "main": "dist/index.cjs",
    "module": "dist/index.mjs",
    "types": "dist/index.d.ts",
    "exports": {
        ".": {
            "import": "./dist/index.mjs",
            "require": "./dist/index.cjs",
            "types": "./dist/index.d.ts"
        },
        "./plugins": {
            "import": "./dist/plugins.mjs",
            "require": "./dist/plugins.cjs"
        }
    },
    "sideEffects": false,
    "files": ["dist"]
}

构建配置（Vite） ​

typescript
// vite.config.ts
import { defineConfig } from 'vite';
import dts from 'vite-plugin-dts';

export default defineConfig({
    build: {
        lib: {
            entry: './src/index.ts',
            name: 'MySDK',
            formats: ['es', 'cjs', 'umd'],
            fileName: (format) => `index.${format === 'es' ? 'mjs' : format === 'cjs' ? 'cjs' : 'umd.js'}`,
        },
        rollupOptions: {
            external: ['react'], // 外部依赖不打包
        },
    },
    plugins: [dts()], // 自动生成类型声明
});

面试回答要点 ​

"设计前端 SDK 要关注五个原则：Tree Shaking 友好（ESM + sideEffects: false）、多格式输出（ESM/CJS/UMD）、零依赖、类型安全、体积控制。package.json 中用 exports 字段配置条件导出，构建工具用 Vite 的 lib 模式，配合 vite-plugin-dts 自动生成类型声明。核心功能保持精简，扩展功能通过子路径导出实现按需加载。"

问题 12：RSC（React Server Components）对构建的影响？ ​

RSC 核心概念 ​

React Server Components 允许组件在服务端渲染，不打包到客户端 bundle。

对构建的影响 ​

传统 ReactRSC
所有组件打包到客户端服务端组件不打包到客户端
客户端 bundle 体积大客户端 bundle 显著减小
数据获取在客户端 useEffect数据获取在服务端直接进行
需要 API 路由服务端组件直接访问数据库

构建流程变化 ​

传统构建：
组件代码 → 打包 → 发送到客户端 → 客户端渲染

RSC 构建：
服务端组件 → 服务端执行 → 生成 RSC Payload → 流式传输到客户端
客户端组件 → 正常打包 → 客户端渲染

面试回答要点 ​

"RSC 对构建的最大影响是服务端组件不打包到客户端 bundle，显著减小客户端体积。构建流程上，服务端组件在服务端直接执行，生成 RSC Payload 流式传输；客户端组件仍正常打包。这要求构建工具支持 RSC 协议，如 Next.js 的 App Router 和 Vite 的 RSC 插件。2026 年 RSC 正在从 Next.js 向更多框架普及。"

问题 14：WebAssembly 在构建中的应用？ ​

应用场景 ​

场景说明
Rust 工具链swc、Rolldown、Turbopack 都用 Rust + WASM
复杂计算图片处理、视频编解码、加密算法
遗留代码复用C/C++ 库编译到 WASM 在浏览器运行

构建工具中的 WASM ​

Babel (JS) → SWC (Rust + WASM) → 10-100x  faster
Rollup (JS) → Rolldown (Rust) → 10x faster
Terser (JS) → esbuild (Go) → 10-100x faster

面试回答要点 ​

"WebAssembly 在构建中的应用主要体现在 Rust 工具链上，如 swc、Rolldown 将 Rust 编译到 WASM 在 Node.js 中运行，获得接近原生的性能。2026 年几乎所有主流构建工具都在用 Rust/Go 重写，通过 WASM 或原生绑定提供 JavaScript API。对于业务代码，WASM 适合复杂计算场景如图片处理、加密算法，将 C/C++ 库编译到浏览器运行。"
