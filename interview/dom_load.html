DOM 加载 ​

这是一份由浅入深的问题清单，覆盖从基础概念到性能优化的完整路径。建议你先独立思考每个问题，再对照我的解析。

第一部分：基础概念（建立认知框架） ​

Q1：从输入 URL 到页面完整显示，DOM 经历了哪几个关键的生命周期阶段？ ​

解析：

大致分为 6 个阶段：

  1. 导航（Navigation） - 发起请求，开始加载
  2. 解析HTML（Parsing） - 浏览器接收字节流 → 解码为字符 → 词法分析生成 Token → 构建 DOM 树
  3. 子资源加载（Subresource Loading） - 遇到 <link>、<script>、<img> 等发起额外请求
  4. DOM 就绪（DOM Ready） - DOM 树构建完成，但外部资源（图片、样式表等）可能还在加载
  5. 页面加载完成（Load） - 所有资源（图片、样式、子框架等）都加载完毕
  6. 卸载（Unload） - 离开页面时触发清理

理解这个顺序，是掌握所有 DOM 加载事件的基础。

第二部分：关键事件解析（理解触发时机） ​

Q2：DOMContentLoaded 和 load 事件的核心区别是什么？哪个先触发？ ​

解析：

  • DOMContentLoaded：先触发。当 HTML 被完全解析，DOM 树构建完成后立即触发，不等待图片、样式表、iframe 等外部资源。
  • load：后触发。等待所有依赖（样式表、图片、音频、视频、子框架等）全部加载完成后才触发。

简单记忆：

  • DOMContentLoaded → DOM 立即可用
  • load → 所有内容都可见可用了

示例验证：

javascript
document.addEventListener('DOMContentLoaded', () => {
    console.log('DOM ready'); // 先输出
});
window.addEventListener('load', () => {
    console.log('All loaded'); // 后输出
});

Q3：readystatechange 事件与 DOMContentLoaded、load 之间有什么关系？ ​

解析：

document.readyState 有三个值，顺序固定：

状态含义对应的事件时机
loading正在加载 HTML初始状态
interactive解析完成，DOM 可交互与 DOMContentLoaded 时机相同
complete所有资源加载完成与 load 时机相同
javascript
document.onreadystatechange = () => {
    if (document.readyState === 'interactive') {
        console.log('DOM ready'); // 几乎等同于 DOMContentLoaded
    }
    if (document.readyState === 'complete') {
        console.log('Page complete'); // 等同于 load
    }
};

注意：DOMContentLoaded 更加语义化，推荐优先使用，但 readystatechange 可以更精细地检测状态变化。

第三部分：脚本行为与干扰（理解阻塞机制） ​

Q4：<script> 标签放在 <head> 和 <body> 底部，对 DOM 加载和事件触发有什么不同影响？ ​

解析：

关键在于 脚本默认会阻塞 HTML 解析。

情况1：放在 <head> 中（不加特殊属性）

  • 遇到 <script> 时，HTML 解析暂停
  • 下载并执行 JS（期间 DOM 构建停滞）
  • 执行完毕后才继续解析
  • 后果：DOMContentLoaded 会被延迟到该脚本执行完后才触发

情况2：放在 </body> 前

  • 几乎整个 HTML 已解析完才遇到脚本
  • DOM 构建基本完成，延迟很小
  • DOMContentLoaded 等待该脚本执行完后触发（但如果脚本异步/延迟，行为不同）

验证方法：

html
<!-- 会阻塞解析约2秒 -->
<script src="slow-script.js"></script>
<!-- 此后的DOM元素需要等脚本执行完才能解析 -->

Q5：async 和 defer 属性如何改变脚本的加载和执行时机？它们对 DOMContentLoaded 有什么影响？ ​

解析：

属性加载时机执行时机对 DOMContentLoaded 的影响
无（普通）遇到即加载立即执行，阻塞解析延迟触发，等它执行完
defer后台并行加载DOM 解析完成后、DOMContentLoaded 之前顺序执行延迟到脚本执行后（但脚本不会阻塞解析）
async后台并行加载加载完成后立即执行（可能阻塞解析）不确定，可能在之前或之后

关键区别图：

普通脚本:   HTML解析[=====停==停==停=====]→DOMContentLoaded
                 ↑下载并立即执行两次

defer:      HTML解析[==================]→执行defer脚本→DOMContentLoaded
                 ↑后台下载，不阻塞

async:      HTML解析[=======停=====]→DOMContentLoaded（时机不确定）
                 ↑下载完立即执行，可能打断解析

最佳实践：

  • 大多数脚本：用 defer，不阻塞解析，且保证执行顺序
  • 独立第三方脚本（如统计代码）：用 async

第四部分：高级场景（处理动态内容和性能） ​

Q6：动态添加的 <script>（用 createElement('script')）会触发 DOM 加载事件吗？如何监听它的加载完成？ ​

解析：

动态脚本默认是 async 行为：

  • 下载过程中不阻塞其他操作
  • 加载完成后立即执行

不会影响 DOMContentLoaded 的时机 —— 因为它是在 DOM 构建完成后才被添加的。

监听动态脚本完成：

javascript
const script = document.createElement('script');
script.src = 'dynamic.js';
script.onload = () => {
    console.log('动态脚本加载并执行完成');
};
script.onerror = () => console.error('加载失败');
document.head.appendChild(script);

注意：onload 在脚本下载并执行完毕后触发。

Q7：document.readyState 已经是 'complete' 之后，再添加 DOMContentLoaded 或 load 监听器会触发吗？ ​

解析：

不会 —— 这两个事件只会在状态变为目标值时触发。如果状态已经过去，新添加的监听器永远不会执行。

安全写法：

javascript
function onReady(callback) {
    if (document.readyState === 'loading') {
        document.addEventListener('DOMContentLoaded', callback);
    } else {
        callback(); // 已经是 'interactive' 或 'complete'，直接执行
    }
}

同理，window.onload 也需要在页面未完成前设置才有意义。

Q8：如何处理图片、字体、异步数据的“真正加载完成”？load 事件不够用时该怎么办？ ​

解析：

load 事件只保证“浏览器原生资源”（图片、CSS、子框架）加载完，但：

  • ❌ 不会等待懒加载的图片（loading="lazy"）
  • ❌ 不会等待通过 fetch() 请求的动态数据
  • ❌ 不会等待 Web 字体加载（只是开始渲染，可能用 fallback）

解决方案：

  1. 等待特定图片/资源：
javascript
const img = document.querySelector('img');
img.complete; // 检查是否已加载
// 或监听 img.onload
  1. 等待 Web 字体：
javascript
document.fonts.ready.then(() => {
    console.log('所有字体加载完成');
});
  1. 等待动态数据：
javascript
// 手动 Promise 控制
await fetch('/api/data');
renderData();
// 此时才算“数据加载完成”
  1. 等待所有懒加载图片：
javascript
const lazyImages = [...document.querySelectorAll('img[loading="lazy"]')];
await Promise.all(
    lazyImages.map((img) => (img.complete ? Promise.resolve() : new Promise((resolve) => (img.onload = resolve)))),
);

第五部分：整合与最佳实践（实战应用） ​

Q9：请用时间线图总结 document.readyState、DOMContentLoaded、load、async/defer 脚本、图片加载的完整顺序。 ​

解析：

时间轴 →

HTML 解析开始 [状态: loading]
    │
    ├─ 遇到普通脚本 → 阻塞等待下载+执行
    │
    ├─ 遇到 defer 脚本 → 后台下载，不阻塞
    │
    ├─ 遇到 async 脚本 → 后台下载，下载完立即执行（可能阻塞）
    │
    ├─ HTML 解析完成 [状态: interactive]
    │           │
    │           └─ 执行所有 defer 脚本（顺序）
    │           └─ 触发 DOMContentLoaded 事件
    │
    ├─ 加载图片、样式表等外部资源（异步并行）
    │           │
    │           └─ 所有资源加载完成 [状态: complete]
    │           └─ 触发 load 事件
    │
页面完全可交互

Q10：实际开发中，基于 DOM 加载状态的最佳实践有哪些？ ​

解析：

1. 操作 DOM 前必须等待 DOMContentLoaded（或更早检查）

javascript
// 错误 ❌
document.querySelector('#app').innerHTML = '...'; // 可能还没解析

// 正确 ✅
document.addEventListener('DOMContentLoaded', () => {
    document.querySelector('#app').innerHTML = '...';
});

2. 需要测量尺寸、图片信息时，用 load

javascript
window.addEventListener('load', () => {
    const width = document.querySelector('img').naturalWidth; // 此时可靠
});

3. 给用户展示 loading 动画的最佳时机

javascript
// 开始 loading
document.body.classList.add('loading');
// DOM 就绪后可先渲染框架
document.addEventListener('DOMContentLoaded', () => renderSkeleton());
// 完全加载完成去掉 loading
window.addEventListener('load', () => document.body.classList.remove('loading'));

4. 避免阻塞首屏渲染

  • 非关键脚本一律加 defer 或 async
  • 内联关键 CSS，异步加载非关键 CSS
  • 首屏不需要的图片加 loading="lazy"

5. 使用现代替代方案

html
<!-- 原生 defer 替代 jQuery 的 $(document).ready -->
<script defer src="main.js"></script>

这些问题覆盖了从入门到高级的核心要点。如果你能清晰回答每一个，就掌握了前端 DOM 加载事件的全貌。

TIP

  • DOMContentLoaded 会等待所有 defer 脚本下载并执行完成后，才会触发。
  • defer 只是不阻塞 HTML 解析过程，但会阻塞 DOMContentLoaded 的触发时机
  • 多个 defer 脚本按在 HTML 中的顺序执行（不像 async 乱序）
  • 如果 defer 脚本下载失败，仍然会触发 DOMContentLoaded（不会无限等待）
