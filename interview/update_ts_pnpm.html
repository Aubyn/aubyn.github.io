TypeScript 和 PNPM 升级 ​

TypeScript 升级到 7.x ​

TypeScript 从 5.x 到 7.x 跨度较大，其中最核心的变化是：v6.0 是一个重大调整版本，淘汰了大量遗留配置；而 v7.0 则带来了全新的 Go 原生编译器，性能提升巨大，但迁移主要是配置层面的，代码层面改动很小。

下面是各版本的关键变化和升级注意事项。

🚀 核心亮点：为什么值得升级？ ​

版本关键变化与亮点性能与体验提升
TypeScript 6.0为 7.0 做准备的“桥梁版本”
• 引入新标准库 API（如 Temporal）
• 默认配置更严格（strict: true）
• 弃用大量旧配置，为 7.0 清理环境		构建性能提升不明显
TypeScript 7.0全新 Go 原生编译器 tsgo
• 编译器从 JavaScript 重写为 Go，速度提升约 10 倍
• 编辑器启动更快，内存占用更低
• 代码和配置与 6.0 完全兼容		构建速度提升 10 倍左右，大型项目效果更显著

⚠️ 升级路径与破坏性改动 ​

升级时主要面临的挑战来自于 TypeScript 6.0。强烈建议先将项目升级到 6.0 并解决所有问题，再平稳过渡到 7.0。

1. tsconfig.json 配置项变更 ​

这是升级到 6.0 时最容易遇到问题的地方，以下是几个重大变更：

  • moduleResolution

    • 变化："node" 选项已废弃（在 7.0 中移除）。
    • 迁移指南：需根据项目类型更新：
      • Node.js 项目：使用 "nodenext" 或 "node16"。
      • 使用构建工具的项目：使用 "bundler"。
    • 参考示例：一个使用 ESM 和路径别名（如 @/）的项目，迁移后配置如下：
      json
      {
    "compilerOptions": {
        "module": "ESNext",
        "moduleResolution": "bundler", // 从 "node" 改为此项
        "paths": {
            "@/*": ["./src/*"] // 路径前需加 "./"
        }
    }
}

  • baseUrl

    • 变化：已废弃（在 7.0 中移除）。
    • 迁移指南：直接删除此配置，并将 paths 中的路径改为相对于 tsconfig.json 的显式路径（如上例所示）。

  • types

    • 变化：默认值从“自动引入 node_modules/@types 下所有包”变为 []（空数组）。
    • 迁移指南：需要在 tsconfig.json 中手动列出项目所需的所有全局类型声明包。
      json
      {
    "compilerOptions": {
        "types": ["node", "jest"] // 显式声明你需要的类型
    }
}

  • strict

    • 变化：现在默认为 true。
    • 迁移指南：如果你的项目之前未开启严格模式，升级后可能会遇到大量类型错误。可以考虑先显式设置为 false 再逐步修复。

  • target

    • 变化：
      • 默认值变为 ES2025。
      • ES5 编译目标已废弃。
    • 迁移指南：如果仍需支持旧版浏览器，需继续使用 Babel 等工具进行转译。

  • outFile

    • 变化：已移除。
    • 迁移指南：使用打包工具（如 esbuild、Webpack、Rollup）代替。

  • import 断言

    • 变化：旧的 assert 语法已废弃，需使用新的 with 语法。

    • 迁移指南：

      typescript
      // ❌ 旧语法
import data from './data.json' assert { type: 'json' };

// ✅ 新语法
import data from './data.json' with { type: 'json' };

2. 代码层面的类型检查加强 ​

  • 函数和方法推断改进：在 v6.0 中，某些涉及复杂泛型的方法调用推断更准确，但也可能要求你提供更明确的类型注解。
  • 命名空间：使用 module Foo {} 声明命名空间现在是硬错误，必须改为 namespace Foo {}。
  • 更严格的空值/真值检查：编译器现在能识别出更多永远为真或假的逻辑，并报错，有助于发现潜在 bug。

📋 分步升级检查清单 ​

建议按以下步骤执行升级，确保平滑过渡：

  1. 升级至最新 5.x 版本：先升级到最新的 TypeScript 5.x 版本（如 5.9.x），修复所有错误。
  2. 更新 tsconfig.json：
    • 替换 moduleResolution。
    • 删除 baseUrl，并更新 paths。
    • 添加 types 数组，列出所需全局类型。
    • 检查 target 和 strict 的设置。
    • 移除 outFile 等废弃选项。
  3. 代码迁移：
    • 将所有 import assert 替换为 import with。
    • 将 module Foo 声明改为 namespace Foo。
    • 根据新的类型检查结果修复代码。
  4. 升级到 TypeScript 6.0：执行 npm install -D typescript@6。运行类型检查 (npx tsc --noEmit) 和测试，修复所有报错。
  5. （可选）预览 TypeScript 7.0：安装预览版或通过 VS Code 扩展体验 tsgo 的性能提升，确保无兼容性问题。

TypeScript 7.0 是一个值得期待的版本，尤其是那 10 倍的性能提升。希望这份清单能帮助你顺利升级！

PNPM 升级到 11.x ​

从 pnpm 10.x 升级到 11.x 是一个变动非常大的主版本。它不是一个常规的功能更新，而是对配置系统、安全模型和底层架构的一次彻底重构。

最大的影响体现在三个方面：配置文件大变样（.npmrc 功能被拆分）、全新的供应链安全默认值（新装包会自动延迟一天）、以及 Node.js 版本要求提高。

这份清单会帮你系统性地梳理所有注意点，避免升级后出现"配置失效"或"安装卡住"的情况。

🚀 pnpm 11.x 核心升级亮点 ​

  • 更强的供应链安全：默认启用 minimumReleaseAge: 1440（新包发布24小时内无法安装），并阻止"外来的"子依赖，有效防御短窗口的供应链攻击。
  • 全新的配置模型：.npmrc 现在仅用于认证和注册表配置，所有 pnpm 专属设置必须迁移到 pnpm-workspace.yaml。
  • 原生实现发布命令：pnpm publish、login 等不再依赖 npm CLI，完全原生实现。
  • 更快更现代的底层：使用 SQLite 数据库（Store v11）和 undici 网络库，存储性能和网络效率显著提升。
  • 全局安装隔离：每个全局包拥有独立的虚拟存储空间，避免依赖冲突。
  • 新增实用命令：pnpm sbom（生成SBOM）、pnpm clean、pnpm peers check 等。

⚠️ 破坏性改动与迁移指南 ​

升级时主要挑战来自配置迁移和环境变更。官方提供了一个自动化代码转换工具，可以完成大部分机械性工作，建议优先使用。

bash
# 方法一：使用 pnpx 一次性运行
pnpx codemod run pnpm-v10-to-v11

# 方法二：全局安装后运行
pnpm add --global codemod
codemod run pnpm-v10-to-v11

以下是代码转换无法解决的几大关键变更，需要手动处理：

1. 环境要求升级 ​

  • Node.js 版本要求：需要 Node.js 22 或更高版本，不再支持 18、19、20、21。
  • 包管理器自身：pnpm 本身现在以 纯 ESM 格式分发。
  • 二进制文件：独立可执行文件需要 glibc 2.27+。

2. 配置文件重构（风险最高） ​

pnpm 11 彻底改变了配置的读取方式：

变更项升级前 (v10)升级后 (v11)手动检查要点
配置位置分散在 .npmrc 和 package.json 的 pnpm 字段统一到 pnpm-workspace.yaml (项目级) 或 ~/.config/pnpm/config.yaml (全局)确保根目录下存在 pnpm-workspace.yaml，并删除了 package.json 中的 pnpm 字段和旧的 .npmrc 非认证配置。
环境变量支持 npm_config_*仅支持 pnpm_config_*全面排查 CI 脚本、Dockerfile 等，将 npm_config_ 全局替换为 pnpm_config_。
仓库配置使用 .npmrc 中的 @scope:registry=使用 pnpm-workspace.yaml 中的 registries 字段按以下格式迁移注册表配置：
registries:
default: https://registry.npmjs.org/
"@my-org": https://private.example.com/
项目级配置各子项目拥有独立 .npmrc统一在 pnpm-workspace.yaml 的 packageConfigs 下配置这是代码转换无法自动完成的操作，需要手动将每个子项目的特殊配置（如 save-exact）写入 packageConfigs 对象中。

3. 安全与脚本控制变更 ​

新版本默认安全策略趋严，可能导致原本正常的依赖安装失败：

变更项升级前 (v10)升级后 (v11)
构建脚本onlyBuiltDependencies, neverBuiltDependencies 等allowBuilds 映射表（显式允许特定包运行脚本）
包年龄限制minimumReleaseAge 默认 0（无限制）默认 1440 分钟（新包发布后需等待24小时）
外来依赖blockExoticSubdeps 默认 false默认 true（阻止来自 Git/Tarball 的间接依赖）

如果升级后安装依赖时报错，请检查 allowBuilds 配置是否正确，并确认 minimumReleaseAge 设置符合你的需求。

4. 全局安装与 Link 命令变更 ​

  • pnpm install -g 被移除：必须使用 pnpm add -g <pkg> 来安装全局包。
  • pnpm link 行为收紧：
    • 不再支持链接全局存储中的包，必须使用相对路径（如 pnpm link ./my-pkg）。
    • 移除 --global 参数，如需将本地包注册为全局命令，使用 pnpm add -g .。
  • 脚本与内置命令重名：如果 package.json 中定义了名为 clean、setup、deploy 或 rebuild 的脚本，执行 pnpm clean 将优先运行脚本而非内置命令。如需强制运行内置命令，请使用 pnpm pm clean。

5. 审计与补丁相关变更 ​

  • pnpm audit 迁移到 GHSA：由于 npm registry API 变更，漏洞过滤不再支持 CVE ID，改用 GHSA ID。需要将 auditConfig.ignoreCves 替换为 auditConfig.ignoreGhsas，并更新其中的 ID 值。
  • 补丁失败处理：ignorePatchFailures 选项已移除，任何补丁应用失败都会直接抛出错误，必须手动修复或移除补丁。
  • pnpm server 命令被移除：无替代方案。

📋 分步升级检查清单 ​

  1. 准备与备份

    • 提交所有未提交的代码（特别是 .npmrc、package.json、pnpm-workspace.yaml 和 pnpm-lock.yaml）。
    • 确保团队 Node.js 版本已统一升级到 22.x 或更高。

  2. 运行自动迁移

    bash
    pnpx codemod run pnpm-v10-to-v11
    • 检查命令执行日志，确认迁移了哪些内容。

  3. 手动修复配置

    • 检查并修改根目录及各级子目录下的配置文件。
    • 检查 .npmrc：确认其中仅包含认证 token 和 registry 地址。
    • 更新 pnpm-workspace.yaml：
      • 确认 allowBuilds 包含了所有需要运行 postinstall 脚本的包（如 esbuild, electron, vue-demi 等）。
      • 如需即时安装新包（不等待24小时），可临时设置 minimumReleaseAge: 0，或通过 minimumReleaseAgeExclude 指定白名单。
      • 迁移自定义注册表配置到 registries 字段。
    • 检查 CI 脚本：将环境变量 npm_config_ 前缀替换为 pnpm_config_。

  4. 执行安装与验证

    • 删除 node_modules 和 pnpm-lock.yaml（由于 Store 版本升级到 v11，全新安装可避免索引问题）。
    • 运行 pnpm install。
    • 运行 pnpm run build 和 pnpm test 进行完整验证。

如果你在走完这套流程后遇到了具体的报错，比如 ERR_PNPM_MISSING_TIME 或 Build script forbidden，可以告诉我具体的错误信息，我来帮你排查。
