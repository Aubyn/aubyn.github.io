k3s nerdctl ​

我来为你设计一套由浅入深的 nerdctl 学习路径。首先提出关键问题，然后逐一解析。

逐题解析 ​

问题 1：nerdctl 是什么？它和 Docker CLI 有什么关系？ ​

核心定位：nerdctl 是 containerd 的命令行客户端，由 containerd 社区官方维护，设计目标是提供与 Docker CLI 几乎一致的命令体验。

关键理解：

  • 架构关系：Docker CLI → Docker Daemon → containerd → runc；而 nerdctl 直接对接 containerd，跳过了 Docker Daemon 这一层
  • 兼容性：nerdctl 的绝大多数命令（run, ps, exec, logs, build 等）与 Docker CLI 语法一致，学习成本极低
  • 优势：更轻量、无守护进程依赖、直接利用 containerd 的全部能力（包括 k3s 内置的 containerd）

一句话总结：如果你会 Docker CLI，你已经会了 80% 的 nerdctl。

问题 2：如何在本地 k3s 环境中安装和配置 nerdctl？ ​

安装方式（推荐二进制）：

bash
# 下载最新版（以 v2.x 为例，建议去 GitHub 查看最新 release）
curl -LO https://github.com/containerd/nerdctl/releases/download/v2.0.0/nerdctl-2.0.0-linux-amd64.tar.gz
tar Cxzvvf /usr/local/bin nerdctl-2.0.0-linux-amd64.tar.gz

# 验证
nerdctl --version

k3s 环境的关键配置：

k3s 默认使用自己管理的 containerd，socket 路径与标准 containerd 不同：

bash
# k3s 的 containerd socket 路径
export CONTAINERD_ADDRESS=/run/k3s/containerd/containerd.sock

# 或者建立 nerdctl 配置文件
mkdir -p ~/.config/nerdctl
cat > ~/.config/nerdctl/nerdctl.toml <<EOF
address = "/run/k3s/containerd/containerd.sock"
namespace = "k8s.io"  # k3s 默认使用 k8s.io 命名空间
EOF

验证连接：

bash
# 查看 k3s 中的镜像（就是 k3s 拉取的所有镜像）
nerdctl -n k8s.io images

# 查看 k3s 中的容器
nerdctl -n k8s.io ps -a

⚠️ 重要：k3s 的 containerd 使用 k8s.io 命名空间隔离 Kubernetes 资源，默认不加 -n 会看不到东西。

问题 3：如何用 nerdctl 完成镜像的拉取、查看和删除？ ​

bash
# 拉取镜像（语法与 docker pull 完全一致）
nerdctl pull nginx:alpine

# 查看本地镜像
nerdctl images

# 查看镜像详情
nerdctl image inspect nginx:alpine

# 删除镜像
nerdctl rmi nginx:alpine

# 清理悬空镜像
nerdctl image prune

与 k3s 结合的场景：

bash
# 手动拉取镜像到 k3s 命名空间，供 Pod 直接使用
# 这样 Pod 创建时就不需要再从仓库拉取
nerdctl -n k8s.io pull myregistry.com/myapp:v1.0

# 查看 k3s 已缓存的镜像
nerdctl -n k8s.io images | grep myapp

问题 4：如何用 nerdctl 运行、停止、查看日志和管理容器？ ​

bash
# 运行容器（与 docker run 几乎一致）
nerdctl run -d --name my-nginx -p 8080:80 nginx:alpine

# 查看运行中的容器
nerdctl ps

# 查看所有容器（包括停止的）
nerdctl ps -a

# 查看日志
nerdctl logs -f my-nginx

# 进入容器
nerdctl exec -it my-nginx sh

# 停止 / 启动 / 删除
nerdctl stop my-nginx
nerdctl start my-nginx
nerdctl rm my-nginx

# 强制删除运行中的容器
nerdctl rm -f my-nginx

与 k3s 结合：

bash
# 查看 k3s 创建的 Pod 容器（容器名通常是 k8s 生成的长串）
nerdctl -n k8s.io ps

# 进入 k3s 中某个 Pod 的容器调试
# 先找到容器 ID
nerdctl -n k8s.io ps | grep my-pod-name
# 再 exec 进去
nerdctl -n k8s.io exec -it <container-id> sh

问题 5：nerdctl 如何与 k3s 内置的 containerd 协同工作？ ​

这是 nerdctl + k3s 的核心价值点：

场景命令示例
查看 k3s 镜像缓存nerdctl -n k8s.io images
预热镜像（节点提前拉取）nerdctl -n k8s.io pull <image>
导出 k3s 中的镜像nerdctl -n k8s.io save -o backup.tar <image>
导入镜像到 k3snerdctl -n k8s.io load -i backup.tar
查看 Pod 容器详情nerdctl -n k8s.io inspect <container-id>
直接调试 Pod 容器nerdctl -n k8s.io exec -it <id> sh
查看容器资源使用nerdctl -n k8s.io stats

命名空间机制理解：

  • containerd 使用命名空间隔离不同用户的资源
  • k3s/Kubernetes 使用 k8s.io 命名空间
  • nerdctl 默认使用 default 命名空间
  • 操作 k3s 资源时必须加 -n k8s.io

问题 6：如何管理容器网络和持久化数据卷？ ​

网络管理：

bash
# 查看网络（nerdctl 默认使用 containerd 的 CNI）
nerdctl network ls

# 创建自定义网络
nerdctl network create my-net

# 运行容器指定网络
nerdctl run -d --net my-net --name app1 nginx:alpine
nerdctl run -d --net my-net --name app2 nginx:alpine
# app1 和 app2 可以通过容器名互相访问

卷管理：

bash
# 创建卷
nerdctl volume create my-data

# 运行容器挂载卷
nerdctl run -d -v my-data:/data --name my-app nginx:alpine

# 查看卷
nerdctl volume ls

# 查看卷详情
nerdctl volume inspect my-data

k3s 中的注意事项：

  • k3s 使用自己的 CNI 插件（通常是 Flannel 或 Cilium），nerdctl -n k8s.io 看到的网络由 Kubernetes 管理
  • 卷在 k3s 中对应 PV/PVC 机制， nerdctl 层面直接操作的是底层目录

问题 7：nerdctl 支持哪些 Docker 不原生支持的高级功能？ ​

功能说明
懒拉取（Lazy Pulling）nerdctl run --snapshotter=stargz 支持 eStargz 镜像，启动前无需完整拉取
加密镜像原生支持 OCI 加密镜像的拉取和运行
多租户命名空间直接利用 containerd 的命名空间实现硬隔离
无根模式（Rootless）更完善的无根容器支持，无需 root 权限运行
Compose 支持nerdctl compose up 兼容 Docker Compose 文件
IPFS 镜像分发支持从 IPFS 网络拉取镜像
Kubernetes CRI 直通可以直接操作 CRI 接口管理的容器（即 k8s Pod）

Compose 示例：

bash
# 与 docker-compose 用法一致
nerdctl compose -f docker-compose.yml up -d
nerdctl compose -f docker-compose.yml down

问题 8：遇到问题时如何排查和调试？ ​

bash
# 1. 确认 nerdctl 连接的是正确的 containerd
nerdctl info
# 检查 Server Address 是否为 /run/k3s/containerd/containerd.sock

# 2. 查看 containerd 服务状态
systemctl status k3s

# 3. 查看 containerd 日志
journalctl -u k3s -f

# 4. 开启 nerdctl 调试输出
nerdctl --debug-full images

# 5. 检查命名空间是否正确
nerdctl namespace ls

# 6. 查看容器详细事件
nerdctl -n k8s.io events

# 7. 检查 CNI 网络配置
cat /etc/cni/net.d/*.conf

# 8. 镜像拉取失败时检查
# - 确认 /etc/containerd/certs.d/ 下的仓库配置
# - 检查 nerdctl login 状态
