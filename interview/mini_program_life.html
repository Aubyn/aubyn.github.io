小程序生命周期 ​

为了帮助你系统地掌握微信小程序生命周期，我设计了五个关键问题，按照**“由点到面、由表及里、由理论到实践”**的逻辑层层递进。

问题清单 ​

  1. 【分类视角】 微信小程序的生命周期主要分为哪两类？它们各自的核心职责是什么？
  2. 【运行视角】 当用户首次打开小程序，到首页渲染完成，App 与 Page 的生命周期回调函数是按什么顺序执行的？
  3. 【开发视角】 onLoad、onShow、onReady 是页面生命周期中最重要的三个节点，在实际开发中该如何根据它们的特性分配业务逻辑？
  4. 【交互视角】 用户触发页面跳转时，原页面和新页面的生命周期是如何交替执行的？
  5. 【进阶视角】 自定义组件的生命周期有哪些？它与所在页面的生命周期存在怎样的执行时机关系？

逐个解析 ​

1. 【分类视角】微信小程序的生命周期主要分为哪两类？ ​

解析逻辑： 采用 What（是什么） 的定义逻辑，构建宏观认知。 回答： 小程序的生命周期主要分为 App 应用生命周期 和 Page 页面生命周期。

  • App 生命周期：监听小程序整体的状态，如初始化、切前台、切后台。
    • 关键钩子：onLaunch（初始化）、onShow（显示）、onHide（隐藏）。
  • Page 生命周期：监听具体页面的状态，如加载、显示、渲染完成、卸载。* 关键钩子：onLoad、onShow、onReady、onHide、onUnload。 核心职责区分： App 层面处理全局逻辑（如登录态检查、全局变量），Page 层面处理 UI 渲染与业务交互。

2. 【运行视角】当用户首次打开小程序，App 与 Page 的生命周期回调函数是按什么顺序执行的？ ​

解析逻辑： 采用 程序运行视角（时间轴顺序），还原真实启动流程。 回答： 小程序启动时，遵循“先初始化应用，再加载页面”的原则，但细节上存在交叉。 执行流程：

  1. 用户点击启动。
  2. App.onLaunch 触发（应用初始化）。
  3. App.onShow 触发（应用显示）。
  4. 随后加载首页，触发 Page.onLoad（页面加载）。
  5. Page.onShow（页面显示）。
  6. Page.onReady（页面初次渲染完成）。 代码示例：
javascript
// app.js
App({
  onLaunch() {
    console.log("1. App onLaunch");
  },
  onShow() {
    console.log("2. App onShow");
  },
});
// pages/index/index.js
Page({
  onLoad() {
    console.log("3. Page onLoad");
  },
  onShow() {
    console.log("4. Page onShow");
  },
  onReady() {
    console.log("5. Page onReady");
  },
});

(控制台输出顺序：1 -> 2 -> 3 -> 4 -> 5) ​

3. 【开发视角】onLoad、onShow、onReady 如何分配业务逻辑？ ​

解析逻辑： 采用 黄金思维圈，解释 Why（为什么在这个位置写代码）。 回答： 这三个钩子代表了页面从加载到可视的三个阶段，职责分明：

  • onLoad (加载阶段)：
    • 特性： 一个生命周期内只执行一次。接受上个页面传递的参数 options。
    • 适用场景： 初始化数据、接口请求、获取路由参数。
  • onShow (显示阶段)：
    • 特性： 只要页面出现（如从后台切回前台、从下个页面返回）都会执行。
    • 适用场景： 刷新页面状态（如重新获取购物车数量）、恢复视频播放。
  • onReady (渲染完成阶段)： _ 特性： 页面 UI 首次渲染完成，可以操作界面节点。 _ 适用场景： 设置界面动态效果、创建动画、获取节点信息（createSelectorQuery）。 代码示例：
javascript
Page({
  onLoad(options) {
    // 1. 获取参数，仅此一次
    this.setData({ id: options.id });
    this.fetchData();
  },
  onShow() {
    // 2. 每次可见时刷新状态（例如返回页面时）
    this.updateCartCount();
  },
  onReady() {
    // 3. 渲染完成，绘制画布或动画
    this.drawCircle();
  },
});

4. 【交互视角】页面跳转时，生命周期是如何交替执行的？ ​

解析逻辑： 采用 用户视角交互顺序（场景化分析），对比 navigateTo 与 redirectTo 的区别。 回答： 页面跳转分为保留当前页面（navigateTo）和关闭当前页面（redirectTo）两种情况，生命周期交替顺序截然不同。 场景 A：wx.navigateTo 跳转（保留旧页）

  1. 旧页面触发 onHide（未销毁）。
  2. 新页面触发 onLoad -> onShow -> onReady。 场景 B：wx.redirectTo 跳转（关闭旧页）
  3. 旧页面触发 onUnload（已销毁）。
  4. 新页面触发 onLoad -> onShow -> onReady。 代码示例（模拟跳转）：
javascript
// 旧页面 pageA
Page({
  onHide() {
    console.log("A: onHide -> 暂时休眠");
  },
  onUnload() {
    console.log("A: onUnload -> 彻底销毁");
  },
});
// 新页面 pageB
Page({
  onLoad() {
    console.log("B: onLoad -> 开始工作");
  },
});
// 跳转逻辑
// wx.navigateTo({ url: 'B' }) => 输出: A onHide -> B onLoad
// wx.redirectTo({ url: 'B' }) => 输出: A onUnload -> B onLoad

5. 【进阶视角】自定义组件的生命周期与页面生命周期的时机关系？ ​

解析逻辑： 采用 依赖先后顺序（父子关系），解析组件与页面的耦合点。 回答： 组件也有自己的生命周期（created、attached、ready 等），它们与所在 Page 的生命周期存在严格的“父-子”依赖时序。 执行时机：

  1. Page onLoad -> 页面初始化。
  2. Component created -> 组件实例被创建（此时不能调用 setData）。
  3. Component attached -> 组件挂载到页面节点树（此时可使用 setData）。
  4. Page onShow -> 页面显示。
  5. Page onReady -> 页面渲染完成。
  6. Component ready -> 组件渲染完成。 关键规律： 组件的 attached 发生在页面 onShow 之前，组件的 ready 发生在页面 onReady 之后。 代码示例：
javascript
// 组件 my-component
Component({
  lifetimes: {
    created() {
      console.log("C: created");
    },
    attached() {
      console.log("C: attached");
    },
    ready() {
      console.log("C: ready");
    },
  },
});
// 页面 index (使用了该组件)
Page({
  onLoad() {
    console.log("P: onLoad");
  },
  onReady() {
    console.log("P: onReady");
  },
});
// 执行顺序：P onLoad -> C created -> C attached -> P onReady -> C ready

总结 ​

理解微信小程序生命周期，建议遵循以下逻辑链条：

  1. 分清主体：App 是全局容器，Page 是交互载体。
  2. 跑通流程：从启动到渲染，心中有数。
  3. 细化职责：onLoad 初始化，onShow 做刷新，onReady 做 UI 交互。
  4. 场景推演：跳转、返回、后台切前台，模拟生命周期的变化。
  5. 组合运用：理解组件与页面的父子时序，避免在组件初始化时操作尚未准备好的页面节点。
