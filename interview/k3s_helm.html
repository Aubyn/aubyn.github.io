helm ​

以下是我为你设计的一套 Helm 由浅入深的问题引导体系，共 12 个问题，分为 5 个阶段。每个问题先提出，再给出解析回答，帮助你从 k3s 基础平滑过渡到 Helm 的实战运用。

第一阶段：建立认知（Helm 是什么？为什么需要它？） ​

问题 1：我已经会用 kubectl 部署 YAML 了，Helm 是什么？它解决了什么问题？ ​

解析回答：

Helm 是 Kubernetes 的包管理工具，类比关系如下：

场景传统工具Kubernetes 世界
安装软件apt install nginxhelm install my-nginx bitnami/nginx
软件包.deb / .pkgChart（Helm 包格式）
仓库apt 源 / pacman 源Helm Repository
配置文件/etc/nginx/nginx.confvalues.yaml

Helm 解决的核心痛点：

  1. YAML 爆炸：一个中等应用可能有 10+ 个 YAML 文件（Deployment、Service、Ingress、ConfigMap、Secret、HPA、ServiceAccount...），手动管理混乱
  2. 环境差异：开发/测试/生产环境配置不同（副本数、资源限制、域名），复制粘贴 YAML 容易出错
  3. 版本管理：升级应用时需要追踪改了哪些 YAML，回滚困难
  4. 复用困难：别人写好的最佳实践无法直接复用

Helm 的核心思想：

  • 把一组相关的 K8s 资源打包成一个 Chart
  • 用 模板 + values 实现配置化
  • 用 Release 管理每次安装/升级/回滚

k3s 内置了 Helm Controller，可以直接通过 HelmChart CRD 来管理 Chart，无需额外安装 Helm CLI。

问题 2：Helm 和 k3s 是什么关系？k3s 内置 Helm 了吗？ ​

解析回答：

k3s 和 Helm 的关系：

k3s 本身不包含 Helm CLI，但内置了 Helm Controller（通过 HelmChart CRD 实现）。这意味着：

方式说明适用场景
Helm CLI传统命令行工具，需单独安装开发调试、本地操作
Helm Controllerk3s 内置，通过 YAML 声明式管理 Chart生产环境、GitOps

安装 Helm CLI（推荐先学命令行）：

bash
# Arch Linux
sudo pacman -S helm

# 验证
helm version

k3s 内置的 Helm Controller 用法：

yaml
apiVersion: helm.cattle.io/v1
kind: HelmChart
metadata:
    name: nginx
    namespace: kube-system
spec:
    chart: nginx
    repo: https://charts.bitnami.com/bitnami
    targetNamespace: default
    valuesContent: |-
        service:
          type: NodePort

把这个 YAML kubectl apply 一下，k3s 就会自动安装 nginx Chart。

建议学习路径：先掌握 Helm CLI 的基本概念，再过渡到 k3s 的 Helm Controller 做自动化管理。

第二阶段：快速上手（安装和使用现有 Chart） ​

问题 3：怎么用 Helm 从仓库安装一个现成的应用？以 Nginx 为例演示完整流程。 ​

解析回答：

完整流程：

bash
# 1. 添加官方仓库（类似 pacman -Syu 前需要配置镜像源）
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo update  # 更新索引

# 2. 搜索可用的 Chart
helm search repo nginx

# 3. 查看 Chart 的默认配置
helm show values bitnami/nginx

# 4. 安装 Chart（创建一个 Release）
helm install my-nginx bitnami/nginx \
  --namespace default \
  --create-namespace

# 5. 查看安装状态
helm list
kubectl get pods,svc,ingress

# 6. 获取访问信息
helm status my-nginx
kubectl get svc my-nginx  # 查看暴露的端口

关键概念：

  • my-nginx：这是 Release 名称，代表这次安装的实例
  • bitnami/nginx：仓库名/Chart名
  • 同一个 Chart 可以安装多次（不同 Release 名），比如 helm install my-nginx-2 bitnami/nginx

卸载：

bash
helm uninstall my-nginx

问题 4：安装的 Chart 默认配置不适合我，怎么自定义配置？values.yaml 是什么？ ​

解析回答：

两种方式自定义：

方式 1：命令行覆盖（快速测试）

bash
helm install my-nginx bitnami/nginx \
  --set replicaCount=3 \
  --set service.type=NodePort \
  --set service.nodePorts.http=30080

方式 2：values.yaml 文件（推荐，可版本控制）

创建 my-values.yaml：

yaml
replicaCount: 3

image:
    repository: nginx
    tag: '1.25'
    pullPolicy: IfNotPresent

service:
    type: NodePort
    nodePorts:
        http: 30080

resources:
    limits:
        cpu: 500m
        memory: 256Mi
    requests:
        cpu: 100m
        memory: 128Mi

ingress:
    enabled: true
    hostname: nginx.local

安装时指定：

bash
helm install my-nginx bitnami/nginx -f my-values.yaml

values.yaml 的作用：

  • Chart 作者定义了一套可配置参数
  • 用户通过 values.yaml 覆盖默认值，无需修改 Chart 内部模板
  • 不同环境（dev/staging/prod）用不同的 values 文件即可

查看 Chart 支持的所有配置项：

bash
helm show values bitnami/nginx > default-values.yaml
# 然后在这个文件基础上修改

问题 5：怎么升级已安装的 Chart？怎么回滚到之前的版本？ ​

解析回答：

升级（Upgrade）：

bash
# 修改 values.yaml 后重新应用
helm upgrade my-nginx bitnami/nginx -f my-values.yaml

# 或只改一个参数
helm upgrade my-nginx bitnami/nginx --set replicaCount=5

Helm 会：

  1. 对比新旧配置差异
  2. 应用变更（只改有变化的部分，不会重建未变更的资源）
  3. 记录为新的 Revision

查看版本历史：

bash
helm history my-nginx
# 输出类似：
# REVISION  UPDATED                   STATUS      CHART        APP VERSION
# 1         2026-06-05 10:00:00       deployed    nginx-15.0.0 1.25.0
# 2         2026-06-05 11:00:00       deployed    nginx-15.0.0 1.25.0
# 3         2026-06-05 12:00:00       failed      nginx-15.0.0 1.25.0

回滚（Rollback）：

bash
helm rollback my-nginx 2  # 回滚到 Revision 2
helm rollback my-nginx    # 简写，回滚到上一个版本

前端场景实战：

bash
# v1.0 发版
helm install frontend ./frontend-chart -f values-prod.yaml

# v1.1 更新镜像
helm upgrade frontend ./frontend-chart -f values-prod.yaml --set image.tag=v1.1

# 发现 bug，秒级回滚
helm rollback frontend

问题 6：我想为自己的前端应用创建一个 Helm Chart，目录结构是什么样的？ ​

解析回答：

创建 Chart 脚手架：

bash
helm create frontend-chart

生成的目录结构：

frontend-chart/
├── Chart.yaml          # Chart 元数据（名称、版本、依赖等）
├── values.yaml         # 默认配置值（用户可覆盖）
├── charts/             # 依赖的子 Chart
├── templates/          # K8s 资源模板（核心）
│   ├── _helpers.tpl    # 命名模板/辅助函数
│   ├── deployment.yaml
│   ├── service.yaml
│   ├── ingress.yaml
│   ├── hpa.yaml        # 自动扩缩容
│   └── NOTES.txt       # 安装后提示信息
└── README.md

各文件解析：

Chart.yaml：

yaml
apiVersion: v2
name: frontend-chart
description: A Helm chart for frontend application
type: application
version: 1.0.0 # Chart 版本
appVersion: '1.0.0' # 应用版本

values.yaml（默认值）：

yaml
replicaCount: 1

image:
    repository: my-frontend
    pullPolicy: IfNotPresent
    tag: 'latest'

service:
    type: ClusterIP
    port: 80

ingress:
    enabled: true
    className: traefik
    hosts:
        - host: frontend.local
          paths:
              - path: /
                pathType: Prefix

resources: {}

解析回答：

Helm 使用 Go template 语法，核心符号是 。

调试模板（不实际安装）：

bash
helm template frontend-chart ./frontend-chart -f my-values.yaml

问题 8：怎么管理不同环境的配置？开发、测试、生产环境用不同的 values 文件？ ​

解析回答：

最佳实践：环境隔离的 values 文件

frontend-chart/
├── values.yaml           # 默认值（开发环境基准）
├── values-dev.yaml       # 开发环境覆盖
├── values-staging.yaml   # 测试环境覆盖
└── values-prod.yaml      # 生产环境覆盖

values-prod.yaml 示例：

yaml
replicaCount: 3

image:
    tag: 'v1.2.0'

resources:
    limits:
        cpu: 1000m
        memory: 512Mi
    requests:
        cpu: 250m
        memory: 128Mi

autoscaling:
    enabled: true
    minReplicas: 3
    maxReplicas: 10
    targetCPUUtilizationPercentage: 80

ingress:
    hosts:
        - host: myapp.example.com
          paths:
              - path: /
                pathType: Prefix

安装/升级时指定环境：

bash
# 开发环境
helm install frontend-dev ./frontend-chart -f values-dev.yaml

# 生产环境
helm install frontend-prod ./frontend-chart -f values-prod.yaml

# 升级生产环境
helm upgrade frontend-prod ./frontend-chart -f values-prod.yaml

进阶：使用 helmfile 管理多环境 如果环境很多，可以用 helmfile 工具统一管理。

第四阶段：Chart 依赖与高级功能 ​

问题 9：我的前端应用依赖 Redis/MySQL，Helm 怎么管理这些依赖？ ​

解析回答：

在 Chart.yaml 中声明依赖：

yaml
apiVersion: v2
name: frontend-chart
version: 1.0.0
dependencies:
    - name: redis
      version: '18.x.x'
      repository: 'https://charts.bitnami.com/bitnami'
      condition: redis.enabled # 可通过 values 控制是否启用
    - name: mysql
      version: '9.x.x'
      repository: 'https://charts.bitnami.com/bitnami'
      condition: mysql.enabled

更新依赖：

bash
helm dependency update ./frontend-chart
# 会下载依赖到 charts/ 目录

values.yaml 中配置依赖：

yaml
redis:
    enabled: true
    auth:
        enabled: false # 开发环境简化
    master:
        persistence:
            enabled: false

mysql:
    enabled: true
    auth:
        rootPassword: devpassword
        database: frontend_db

模板中引用依赖服务：

yaml
env:
    - name: REDIS_HOST
      value: '{{ .Release.Name }}-redis-master'
    - name: MYSQL_HOST
      value: '{{ .Release.Name }}-mysql'

安装时自动部署所有依赖：

bash
helm install myapp ./frontend-chart
# 会自动安装 frontend + redis + mysql

问题 10：Helm 的 Hook 是什么？怎么用 Hook 实现数据库迁移？ ​

解析回答：

Hook 机制： Helm 允许在 Release 生命周期的特定阶段执行额外的 K8s 资源（Job），比如：

  • pre-install：安装前执行
  • post-install：安装后执行
  • pre-upgrade：升级前执行
  • post-upgrade：升级后执行
  • pre-delete：删除前执行
  • post-delete：删除后执行

前端场景：数据库迁移 Job

创建 templates/migration-job.yaml：

yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: {{ include "frontend-chart.fullname" . }}-db-migrate
  annotations:
    "helm.sh/hook": pre-upgrade,pre-install  # 安装和升级前执行
    "helm.sh/hook-weight": "-5"              # 执行顺序（越小越早）
    "helm.sh/hook-delete-policy": before-hook-creation,hook-succeeded  # 完成后删除
spec:
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: migrate
          image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
          command: ["npm", "run", "db:migrate"]
          env:
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef:
                  name: {{ include "frontend-chart.fullname" . }}-db
                  key: url

执行流程：

helm upgrade myapp ./frontend-chart
  → 执行 pre-upgrade Hook（数据库迁移 Job）
  → 等待 Job 成功
  → 更新 Deployment
  → 执行 post-upgrade Hook（可选，如缓存预热）

第五阶段：k3s + Helm 生产化实战 ​

问题 11：怎么在 k3s 中用 Helm Controller（HelmChart CRD）实现声明式 Chart 管理？ ​

解析回答：

HelmChart CRD 是 k3s 的原生能力，无需 Helm CLI 即可管理 Chart。

示例：通过 YAML 安装应用

yaml
apiVersion: helm.cattle.io/v1
kind: HelmChart
metadata:
    name: frontend
    namespace: kube-system # HelmChart 资源放在 kube-system
spec:
    chart: frontend-chart
    repo: '' # 本地 Chart 留空
    version: '1.0.0'
    targetNamespace: default
    valuesContent: |-
        replicaCount: 2
        image:
          repository: my-frontend
          tag: "v1.0.0"
        ingress:
          enabled: true
          className: traefik
          hosts:
            - host: frontend.local
              paths:
                - path: /
                  pathType: Prefix

本地 Chart 的部署方式：

bash
# 1. 将 Chart 放到 k3s 的 Chart 目录
sudo mkdir -p /var/lib/rancher/k3s/server/static/charts
sudo cp frontend-chart-1.0.0.tgz /var/lib/rancher/k3s/server/static/charts/

# 2. 应用 HelmChart CRD
kubectl apply -f frontend-helmchart.yaml

查看 Helm Controller 状态：

bash
kubectl get helmchart -n kube-system
kubectl describe helmchart frontend -n kube-system

优势：

  • 完全声明式，适合 GitOps
  • 不需要在 CI/CD 中安装 Helm CLI
  • k3s 自动处理安装/升级/回滚

问题 12：我想实现"Git 提交 → 自动构建 → 自动用 Helm 部署到 k3s"，完整的 CI/CD 流水线怎么搭建？ ​

解析回答：

完整流水线架构：

Git Push → GitHub Actions → Build Image → Push Registry → Update Helm Values → Deploy to k3s

1. Git 仓库结构：

my-frontend/
├── src/                    # 前端源码
├── Dockerfile
├── helm/
│   └── frontend/           # Helm Chart
│       ├── Chart.yaml
│       ├── values.yaml
│       └── templates/
├── .github/
│   └── workflows/
│       └── deploy.yml      # CI/CD 配置
└── values-prod.yaml        # 生产环境配置

2. GitHub Actions 配置（.github/workflows/deploy.yml）：

yaml
name: Build and Deploy

on:
    push:
        branches: [main]
        tags: ['v*']

jobs:
    build:
        runs-on: ubuntu-latest
        steps:
            - uses: actions/checkout@v4

            # 1. 构建并推送镜像
            - name: Docker Login
              uses: docker/login-action@v3
              with:
                  registry: ghcr.io
                  username: ${{ github.actor }}
                  password: ${{ secrets.GITHUB_TOKEN }}

            - name: Docker Build & Push
              run: |
                  VERSION=${GITHUB_REF#refs/tags/}
                  if [ "$VERSION" = "$GITHUB_REF" ]; then VERSION=latest; fi
                  docker build -t ghcr.io/${{ github.repository }}:${VERSION} .
                  docker push ghcr.io/${{ github.repository }}:${VERSION}
                  echo "VERSION=${VERSION}" >> $GITHUB_ENV

            # 2. 更新 Helm values 并部署
            - name: Setup Helm
              uses: azure/setup-helm@v4

            - name: Setup Kubectl
              uses: azure/setup-kubectl@v4

            - name: Configure Kubeconfig
              run: |
                  mkdir -p ~/.kube
                  echo "${{ secrets.KUBECONFIG }}" > ~/.kube/config

            - name: Deploy to k3s
              run: |
                  helm upgrade --install frontend ./helm/frontend \
                    --namespace production \
                    --create-namespace \
                    -f values-prod.yaml \
                    --set image.tag=${{ env.VERSION }} \
                    --wait \
                    --timeout 5m

            - name: Verify Deployment
              run: |
                  kubectl rollout status deployment/frontend -n production
                  kubectl get pods,svc,ingress -n production

3. Arch k3s 机器准备：

bash
# 导出 kubeconfig（把 server 改为公网 IP）
sudo cat /etc/rancher/k3s/k3s.yaml | sed 's/127.0.0.1/<你的公网IP>/' > k3s-config
# 将内容复制到 GitHub Secrets: KUBECONFIG

# 开放防火墙（如果 k3s 在公网）
sudo firewall-cmd --add-port=6443/tcp --permanent
sudo firewall-cmd --reload

4. 进阶：使用 Helm Controller + GitOps（ArgoCD）

yaml
# 在 k3s 中安装 ArgoCD
kubectl create namespace argocd
kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml

# 创建 ArgoCD Application
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: frontend
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/yourname/my-frontend.git
    targetRevision: main
    path: helm/frontend
    helm:
      valueFiles:
        - values-prod.yaml
  destination:
    server: https://kubernetes.default.svc
    namespace: production
  syncPolicy:
    automated:
      prune: true
      selfHeal: true

这样每次 Git 提交，ArgoCD 会自动同步到 k3s，实现真正的 GitOps。
