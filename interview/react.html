关键问题列表（由浅入深） ​

  1. React 和 Vue 3 的核心设计哲学有什么本质不同？
    （比如运行时 vs 编译时优化、数据可变性 vs 不可变性）

  2. React 中如何定义一个组件？函数组件和类组件分别怎么写？它和 Vue 3 的 SFC（单文件组件）以及组合式 API 有什么对应关系？

  3. React 的状态管理：useState 和 useReducer 的工作原理是什么？与 Vue 3 的 ref / reactive 有哪些关键区别？

  4. React 中的副作用和生命周期如何实现？useEffect 与 Vue 3 的 watch、watchEffect、onMounted 等怎么对应？

  5. 跨组件传递数据：React 的 Context API 和 Vue 3 的 provide/inject 有何异同？

  6. 表单处理：React 中的受控组件和非受控组件是什么？和 Vue 的 v-model 有何区别？

  7. 性能优化：React 的 memo、useMemo、useCallback 分别解决什么问题？与 Vue 3 的 computed、watch、v-once 等对比如何？

  8. 如何操作原生 DOM？React 的 ref 与 Vue 3 的 ref（模板引用）对比。

  9. 路由和状态管理：React 生态中的 React Router 和 Zustand / Redux，与 Vue Router 和 Pinia 相比，使用思路有什么不同？

  10. 从 Vue 3 转到 React 最容易踩的坑有哪些？有哪些推荐的迁移思维转变？

逐个解析回答 ​

1. React 和 Vue 3 的核心设计哲学有什么本质不同？ ​

核心区别：运行时 vs 编译时优化，以及数据可变性 vs 不可变性

  • Vue 3：基于响应式代理（Proxy），数据默认是可变的。你直接修改 state.count++，Vue 内部能够追踪变化并触发更新。它通过编译时模板分析和运行时依赖收集来达到精确更新，对开发者更“自动”。

  • React：采用不可变数据流和显式状态更新。你永远不能直接修改状态，必须通过 setState（类组件）或 setXXX（函数组件）替换整个状态值。React 通过对比新旧状态（浅比较或全等）来决定是否重新渲染组件。这种设计使得更新可预测，但需要你手动保证状态不可变。

影响：

  • Vue 中你可以轻松写出 obj.prop = newVal，React 中必须 setObj({ ...obj, prop: newVal })。
  • React 的重新渲染范围更依赖你（memo、useCallback 等），Vue 的依赖追踪更自动化。

一句话总结：Vue 帮你做了更多“魔法”，React 让你更显式地控制数据变化和渲染时机。

2. React 中如何定义组件？与 Vue 3 的 SFC / 组合式 API 如何对应？ ​

React 组件定义：

  • 函数组件（现代推荐）：
    jsx
    function Greeting({ name }) {
    return <div>Hello, {name}!</div>;
}
  • 类组件（旧方式，了解即可）：
    jsx
    class Greeting extends React.Component {
    render() {
        return <div>Hello, {this.props.name}!</div>;
    }
}

与 Vue 3 的对应：

  • Vue 3 的 SFC（.vue 文件）包含 <template>、<script>、<style>。React 没有内置的模板语法，JSX 就是 JavaScript，样式通常用 CSS-in-JS 或单独的 CSS 文件导入。
  • Vue 3 的 组合式 API（setup 函数 + ref/reactive）与 React hooks（useState、useEffect 等）在组织逻辑上非常相似。不同点：
    • Hooks 只能在函数组件最顶层调用，不能在循环/条件中使用；组合式 API 也要求顶层调用，但允许在函数内封装。
    • Vue 的响应式数据需要用 .value 访问（在模板中自动解包），React 的状态直接是值。

本质上：组件 = 返回 JSX 的函数。JSX 是 React.createElement 的语法糖，模板能力完全由 JavaScript 提供。

3. React 状态管理：useState / useReducer 与 Vue 3 的 ref / reactive ​

useState：

jsx
const [count, setCount] = useState(0);
setCount(count + 1); // 替换式更新
  • 状态是不可变对象，每次更新都产生新值。
  • 如果新值依赖旧值，推荐用函数形式：setCount(prev => prev + 1)。

useReducer（适合复杂状态逻辑）：

jsx
const [state, dispatch] = useReducer(reducer, initialState);
// dispatch({ type: 'increment' })

与 Vue 对比：

  • Vue 的 ref 是一个包含 .value 的响应式对象，可直接修改：count.value++。
  • Vue 的 reactive 用于对象/数组，也是直接修改属性：obj.key = newVal。
  • 关键差异：React 的 setter 是替换整个状态；Vue 的修改是原地修改。这导致了：
    • React 中更新对象时必须创建副本（展开运算符或 Immer）。
    • Vue 中直接修改更方便，但跨组件传递响应式对象时需小心失去响应性（用 toRefs 等）。

实践建议：在 React 中养成“永远不要直接修改状态”的习惯，否则组件不会重新渲染。

4. 副作用和生命周期：useEffect 与 Vue 3 的 watch / watchEffect / onMounted ​

useEffect 统一处理所有副作用（DOM 操作、数据请求、订阅等）：

jsx
useEffect(() => {
    // 副作用代码
    return () => {
        /* 清理函数（类似 beforeDestroy） */
    };
}, [dependencies]); // 依赖数组
  • 空依赖 [] → 只在挂载后执行一次（类似 onMounted）。
  • 没有依赖 → 每次渲染后都执行（类似 watchEffect 但时机不同）。
  • 有具体依赖 → 依赖变化时执行（类似 watch 的 immediate: false）。

与 Vue 3 对应：

  • onMounted + onUnmounted 分别对应 useEffect 的空依赖与清理函数。
  • watchEffect 自动收集依赖，类似于 useEffect 不带依赖数组（但 Vue 的 watchEffect 会在依赖变化时立即重新运行，而 useEffect 是在渲染后异步执行，且 React 的渲染可能被批处理）。
  • watch 明确监听某个源，类似于 useEffect 显式传入依赖数组。

重要差异：

  • React 的 useEffect 依赖数组必须穷举所有被使用的外部变量，否则会捕获旧值（闭包陷阱）。Vue 的 watch / watchEffect 会自动追踪依赖。
  • Vue 的生命周期钩子（onBeforeUpdate、onUpdated）在 React 中没有直接对应，通常用 useLayoutEffect（同步执行）或 useEffect 模拟。

技巧：遇到闭包陷阱时，考虑使用 useRef 保存最新值，或正确声明依赖数组。

5. 跨组件传递数据：React Context 与 Vue provide/inject ​

React Context：

jsx
const ThemeContext = React.createContext('light');

function Parent() {
    return (
        <ThemeContext.Provider value="dark">
            <Child />
        </ThemeContext.Provider>
    );
}

function Child() {
    const theme = useContext(ThemeContext); // 消费
    return <div>{theme}</div>;
}

与 Vue 对比：

  • 相同点：都是依赖注入模式，解决 props 逐层传递问题。
  • 不同点：
    • React Context 的 Provider 必须包裹在组件树中，且值改变时，所有 useContext 的组件都会重新渲染（除非使用 memo 优化）。
    • Vue 的 provide / inject 默认不是响应式的，需要配合 ref / reactive 提供响应式值；但 Vue 的注入不会自动触发组件重渲染，只有被注入的响应式数据变化时才会。
    • React 可以创建多个独立的 Context（类似 Vue 的多个 provide 键），但每个 Context 的值是全局的（相对 Provider 子树）。
    • React 没有类似 Vue 中 inject 的默认值和可选“避免嵌套冲突”的机制，需要手动处理。

注意：Context 不适合高频变化的值（如每秒更新），因为会导致大量重渲染，此时应使用状态管理库（如 Zustand、Redux）。

6. 表单处理：受控组件 vs 非受控组件，与 v-model ​

受控组件（推荐）：

jsx
function MyForm() {
    const [value, setValue] = useState('');
    return <input value={value} onChange={(e) => setValue(e.target.value)} />;
}
  • React 自己管理输入框的值，通过 value + onChange 实现“单向数据流 + 事件触发更新”。

非受控组件：

jsx
const inputRef = useRef();
<input defaultValue="initial" ref={inputRef} />;
// 需要时通过 inputRef.current.value 获取值

与 v-model 对比：

  • Vue 的 v-model 是语法糖，等价于 :value="value" @input="value = $event"，自动实现了双向绑定。
  • React 没有内置双向绑定，你必须显式写出 value 和 onChange。这更啰嗦，但逻辑更透明。
  • 对于复杂表单，通常封装自定义 hook 来管理多个字段（类似 useForm 库）。

提示：从 Vue 转来会觉得麻烦，但习惯后你会喜欢这种显式控制。另外 React 支持 onChange 事件，注意它不是原生的 input 事件，而是 React 内部合成的。

7. 性能优化：memo、useMemo、useCallback vs Vue 3 的 computed 等 ​

React 优化工具：

  • React.memo：对组件进行浅比较 props，如果没变化则跳过重新渲染。相当于 Vue 中手动使用 v-once 或 shallowRef，但更接近手动实现 shouldComponentUpdate。
  • useMemo：缓存计算结果，类似 Vue 的 computed。
    jsx
    const expensiveValue = useMemo(() => compute(a, b), [a, b]);
  • useCallback：缓存函数引用，防止子组件因引用变化而重新渲染（常与 memo 配合）。Vue 中没有完全对应的概念（因为 Vue 的依赖追踪更细粒度），但可以用 useDebounceFn 或普通函数（Vue 的更新不依赖函数标识）。

与 Vue 对比：

  • Vue 的 computed 自动缓存且惰性求值，React 的 useMemo 也是缓存，但需要手动声明依赖。
  • Vue 的组件更新粒度更细（因为模板编译 + Proxy），通常不需要像 React 那样大量使用 memo；React 默认是从根到叶子全树重渲染，优化需要手动干预。
  • Vue 3 的 watchEffect 和 watch 没有直接对应 useCallback，因为 Vue 不需要用函数标识来避免子渲染。

经验：在 React 中，不要过早优化。只有当出现性能问题时，再使用 memo + useCallback/useMemo。

8. 操作原生 DOM：React 的 ref vs Vue 3 的 ref（模板引用） ​

React ref：

jsx
const inputRef = useRef(null);
<input ref={inputRef} />;
// 访问：inputRef.current.focus()
  • 也可以使用回调 ref：<input ref={el => (inputEl = el)} />
  • 类组件中可用 createRef。

与 Vue 3 模板引用的异同：

  • 相同点：都通过 .current（Vue 中是 .value）访问 DOM 元素；都支持函数式 ref。
  • 不同点：
    • Vue 中定义 const inputRef = ref(null)，模板中 ref="inputRef"，然后 inputRef.value.focus()。
    • React 的 useRef 不仅仅用于 DOM，还可以保存任意可变值（且改变 .current 不会触发重渲染），相当于 Vue 中的 shallowRef 但非响应式。
    • React 中 ref 转发需要使用 forwardRef 高阶组件，Vue 中默认支持（ref 可以挂在组件上自动传递）。

小技巧：在 React 中，尽量避免用 ref 操作 DOM 来改变样式/内容，应该用状态驱动视图。

9. 路由和状态管理生态对比 ​

功能Vue 3 生态React 生态
路由Vue Router (官方)React Router (事实标准，非官方)
状态管理Pinia (官方推荐)Zustand (轻量)、Redux (重量)、Jotai (原子化)

React Router 与 Vue Router 对比：

  • 概念类似：<Router> (Vue 中是 createRouter)，<Routes> / <Route> (Vue 中是 <RouterView> + 路由配置)。
  • 主要差异：React Router 中每个路由对应的组件就是普通 React 组件，不要求 <router-view> 占位符；嵌套路由通过 <Outlet /> 组件实现，而 Vue 是通过嵌套的 <RouterView>。
  • 路由传参：React 用 useParams、useSearchParams hook；Vue 用 useRoute。

状态管理对比：

  • Pinia：基于 Vue 3 响应式系统，可以直接修改 state，组合式 API 风格。
  • Zustand（推荐代替 Redux）：
    jsx
    const useStore = create((set) => ({
    count: 0,
    inc: () => set((state) => ({ count: state.count + 1 })),
}));
// 组件中：const { count, inc } = useStore();
    与 Pinia 类似的是都鼓励直接修改（但 Zustand 中须用 set 替换不可变更新，而 Pinia 允许 state.count++）。
  • Redux：更严格的不可变更新 + 样板代码多，现在多被 Zustand 取代。

建议：React 初学者从 Zustand 开始，它和 React 的不可变理念匹配，同时很轻量。

10. 从 Vue 3 转到 React 最容易踩的坑及思维转变 ​

常见陷阱：

  1. 直接修改状态
    ❌ state.count++
    ✅ setState({ ...state, count: state.count + 1 })

  2. 忘记 React 组件的重新渲染是“整个函数重新执行”

    • 导致你在组件内定义的函数、对象每次渲染都是新引用，可能破坏 memo 优化。
    • 解决方法：用 useCallback / useMemo 包装。

  3. useEffect 依赖数组缺失导致的无限循环

    jsx
    useEffect(() => {
    setCount(count + 1); // 依赖数组为空，但使用了 count → 每次执行后 count 变化再次触发
}, []); // 错误！应为 [count]

  4. 在条件语句或循环中调用 Hooks
    React 严格要求 Hooks 每次渲染顺序一致，否则会报错。

  5. 混淆 JSX 中的 class 和 className，以及 for → htmlFor。

思维转变：

  • 从“自动追踪”到“显式声明”：Vue 帮你自动收集依赖，React 需要你手动列出依赖（useEffect、useMemo 等）。
  • 从“可变数据”到“不可变数据”：放弃 obj.prop = x 的习惯，拥抱展开运算符或 Immer。
  • 从“模板 DSL”到“纯 JS 表达力”：JSX 只是 JS，你可以直接使用 map、filter、三元运算符，不需要 Vue 的 v-if、v-for。
  • 更频繁地考虑“渲染性能”：因为 React 默认全树重渲染，你需要学习 memo、useCallback。

最后建议：先忽略类组件，直接从函数组件 + Hooks 开始。找一个小项目（如计数器、待办事项）用 React 重写，对比 Vue 的实现，你会很快上手。
