goroutine 常见问题 ​

下面围绕 死锁、数据竞态、errgroup 三个主题，提出由浅入深的关键问题。你可以先尝试自己回答，再对照解析和代码示例。

问题列表 ​

死锁 ​

  1. 什么是死锁？在 goroutine 和 channel 的使用中，最简单的死锁示例是怎样的？
  2. 为什么向无缓冲 channel 发送数据没有对应的接收方会导致死锁？
  3. 下面的代码为什么死锁？如何修复？
    go
    ch1 := make(chan int)
ch2 := make(chan int)
go func() { ch1 <- <-ch2 }()
go func() { ch2 <- <-ch1 }()
<-ch1

数据竞态 ​

  1. 什么是数据竞态？用代码演示一个典型的数据竞态现象。
  2. 如何检测数据竞态？Go 提供了什么工具？
  3. 解决数据竞态的方法有哪些？分别给出代码示例。

errgroup ​

  1. errgroup 是什么？与标准库 sync.WaitGroup 相比有什么优势？
  2. 如何使用 errgroup 同时执行多个任务，并处理第一个出现的错误？
  3. errgroup 如何限制并发数量？请给出一个控制最多同时运行 3 个 goroutine 的例子。
  4. 使用 errgroup 时，子 goroutine 发生 panic 会怎样？如何安全处理？

问题解析与代码示例 ​

1. 什么是死锁？最简单的死锁示例？ ​

解析：
死锁是指两个或多个 goroutine 互相等待对方释放资源，导致所有相关 goroutine 永久阻塞。
最简单的例子：主 goroutine 从无缓冲 channel 接收，但没有其他 goroutine 发送。

代码示例：

go
package main

func main() {
    ch := make(chan int)
    <-ch // 阻塞，等待发送，但永远不会有发送
}

程序会报 fatal error: all goroutines are asleep - deadlock!

2. 为什么向无缓冲 channel 发送数据没有对应的接收方会导致死锁？ ​

解析：
无缓冲 channel 的发送操作会阻塞，直到另一个 goroutine 执行接收。如果发送后没有机会被接收（且没有其他 goroutine 继续运行），就永远无法唤醒，导致死锁。
注意：如果发送发生在主 goroutine，并且没有其他 goroutine 接收，主 goroutine 自己阻塞，程序死锁。

代码示例（死锁）：

go
func main() {
    ch := make(chan int)
    ch <- 42 // 阻塞，没有接收者，且没有其他活跃 goroutine
}

3. 通道循环等待的死锁代码如何修复？ ​

代码分析：

go
ch1 := make(chan int)
ch2 := make(chan int)
go func() { ch1 <- <-ch2 }()
go func() { ch2 <- <-ch1 }()
<-ch1

为什么死锁：

  • 第一个 goroutine 需要从 ch2 接收再发送给 ch1。
  • 第二个 goroutine 需要从 ch1 接收再发送给 ch2。
  • 两者都等待对方先发送，形成循环等待，且没有第三个 goroutine 打破。
  • 主 goroutine 等待 ch1 接收，但 ch1 无人能发送成功。

修复方法：使用一个额外的 goroutine 或缓冲 channel，或者打破依赖关系。简单修复：给某个 channel 加缓冲或提前发送一个值。

修复示例：

go
ch1 := make(chan int, 1) // 加缓冲
ch2 := make(chan int)
go func() { ch1 <- <-ch2 }()
go func() { ch2 <- <-ch1 }()
ch1 <- 0 // 预先塞入一个值，打破循环
<-ch1

4. 什么是数据竞态？典型示例？ ​

解析：
数据竞态发生在多个 goroutine 并发访问同一内存区域，且至少有一个是写操作，没有同步控制。结果是不可预测的。

代码示例：

go
package main

import "fmt"

func main() {
    counter := 0
    for i := 0; i < 1000; i++ {
        go func() {
            counter++ // 非原子操作：读-改-写
        }()
    }
    fmt.Println(counter) // 大概率不是 1000，每次运行可能不同
}

5. 如何检测数据竞态？ ​

解析：
Go 内置了竞态检测器。在编译/运行测试时加上 -race 标志。例如：
go run -race main.go 或 go test -race。
它会记录内存访问，检测到冲突时输出报告并退出。

6. 解决数据竞态的方法及代码示例 ​

解析：
三种主流方法：

  • 互斥锁（sync.Mutex）
  • 原子操作（sync/atomic）
  • 使用 channel 进行通信（不要通过共享内存来通信）

示例 1：Mutex

go
var mu sync.Mutex
var counter int

for i := 0; i < 1000; i++ {
    go func() {
        mu.Lock()
        counter++
        mu.Unlock()
    }()
}

示例 2：原子操作

go
var counter int32
for i := 0; i < 1000; i++ {
    go func() {
        atomic.AddInt32(&counter, 1)
    }()
}

示例 3：channel

go
ch := make(chan int, 1000)
counter := 0
for i := 0; i < 1000; i++ {
    go func() {
        ch <- 1
    }()
}
for i := 0; i < 1000; i++ {
    counter += <-ch
}

7. errgroup 是什么？相比 WaitGroup 的优势？ ​

解析：
errgroup 是 golang.org/x/sync/errgroup 包提供的工具，它在 sync.WaitGroup 基础上增加了错误传播和上下文取消功能。
优势：

  • 可以收集子 goroutine 返回的第一个错误（非 nil 错误）。
  • 配合 context，一个任务出错可以自动取消其他任务。

8. 如何使用 errgroup 执行多个任务并处理第一个错误？ ​

解析：
使用 errgroup.Group，调用 Go(func() error) 启动任务，最后调用 Wait() 等待所有任务并返回第一个非 nil 错误。

代码示例：

go
package main

import (
    "fmt"
    "golang.org/x/sync/errgroup"
    "time"
)

func main() {
    var g errgroup.Group
    g.Go(func() error {
        time.Sleep(1 * time.Second)
        return nil // 成功
    })
    g.Go(func() error {
        time.Sleep(500 * time.Millisecond)
        return fmt.Errorf("task 2 failed")
    })
    if err := g.Wait(); err != nil {
        fmt.Println("First error:", err) // 输出第一个错误
    }
}

9. errgroup 如何限制并发数量？ ​

解析：
使用 errgroup.WithContext 和 SetLimit 方法（Go 1.20+ 支持）。SetLimit(n) 限制同时运行的 goroutine 数量为 n。

示例：限制最多 3 个并发：

go
package main

import (
    "fmt"
    "golang.org/x/sync/errgroup"
    "time"
)

func main() {
    g, ctx := errgroup.WithContext(context.Background())
    g.SetLimit(3) // 最多同时 3 个 goroutine

    for i := 0; i < 10; i++ {
        i := i
        g.Go(func() error {
            // 模拟任务
            time.Sleep(100 * time.Millisecond)
            fmt.Printf("Done task %d\n", i)
            return nil
        })
    }
    if err := g.Wait(); err != nil {
        fmt.Println(err)
    }
}

10. errgroup 中发生 panic 会怎样？如何安全处理？ ​

解析：
默认情况下，子 goroutine 中的 panic 会导致整个程序崩溃（除非在 Go 函数内部 recover）。errgroup 不会自动 recover。
安全处理：在 Go 调用的闭包内部使用 defer recover()，并将 recover 得到的错误作为 error 返回。

示例：

go
var g errgroup.Group
g.Go(func() (err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("panic: %v", r)
        }
    }()
    // 可能 panic 的代码
    panic("something wrong")
    return nil
})
if err := g.Wait(); err != nil {
    fmt.Println("Caught panic as error:", err)
}
