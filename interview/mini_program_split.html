## 背景

我想学习微信小程序的分包相关知识

## 任务

提出一系列关键问题，来引导由浅到深地理解并入手。先提出问题，再逐个解析回答

好的，以下是关于微信小程序分包的一系列关键问题，我会按照由浅入深的顺序提出，然后逐个为你解析解答。

一、基础认知篇 ​

问题 1：什么是微信小程序分包？为什么需要它？ ​

解析：

微信小程序分包是一种代码体积优化机制。小程序有主包体积限制（目前单个包不超过 2MB，总体积不超过 20MB），当项目代码、图片、组件越来越多时，主包很容易超限。分包允许将部分页面和资源拆分成独立的"子包"，用户使用时按需下载，而不是一次性加载全部代码。

核心价值：

  • 解决主包体积超限问题
  • 缩短首次启动时间（只下载主包）
  • 支持按需加载，提升用户体验

问题 2：分包有哪些类型？它们有什么区别？ ​

解析：

微信小程序目前有两种分包类型：

类型说明特点
普通分包最常用的分包方式用户进入分包页面时才下载，可以依赖主包代码
独立分包特殊分包，可独立运行不依赖主包即可启动，适合活动页、广告落地页等

关键区别：

  • 普通分包必须先加载主包才能使用
  • 独立分包可以不下载主包直接启动，但有一些限制（如不能依赖主包组件、不能访问 getApp() 等）

问题 3：主包和分包在 app.json 中是如何配置的？ ​

解析：

配置结构如下：

json
{
    "pages": ["pages/index/index", "pages/logs/logs"],
    "subpackages": [
        {
            "root": "packageA",
            "pages": ["pages/cat/cat", "pages/dog/dog"]
        },
        {
            "root": "packageB",
            "name": "pack2",
            "pages": ["pages/apple/apple", "pages/banana/banana"],
            "independent": true
        }
    ]
}

配置要点：

  • pages：主包页面
  • subpackages：分包配置数组
  • root：分包根目录
  • name：分包别名（用于预下载）
  • independent：是否为独立分包（true 表示是）

二、实践操作篇 ​

问题 4：如何规划哪些页面/资源应该放入分包？ ​

解析：

建议放入分包的：

  • 低频使用的功能页面（如设置页、关于页、帮助中心）
  • 特定业务模块（如电商的订单详情、物流查询）
  • 大型第三方库或组件（只在特定页面使用）
  • 活动页面、推广落地页（可设为独立分包）

建议留在主包的：

  • TabBar 页面（必须放在主包）
  • 全局公共组件和工具
  • 启动时必须用到的核心逻辑

原则： 按业务模块或使用频率划分，不要过度拆分导致管理混乱。

问题 5：分包页面之间如何跳转？分包与主包之间如何跳转？ ​

解析：

跳转方式与普通页面完全一致，使用 wx.navigateTo、wx.redirectTo、wx.switchTab 等 API：

javascript
// 从主包跳转到分包页面
wx.navigateTo({
    url: '/packageA/pages/cat/cat',
});

// 从分包A跳转到分包B
wx.navigateTo({
    url: '/packageB/pages/apple/apple',
});

// 从分包回到主包
wx.navigateTo({
    url: '/pages/index/index',
});

注意：

  • TabBar 页面必须用 wx.switchTab 跳转
  • 分包路径必须以 / 开头，完整写出 root + pages 路径

问题 6：分包之间可以共享组件和代码吗？如何做到？ ​

解析：

分包之间默认是隔离的，但可以通过以下方式共享：

方式说明
主包共享将公共组件、工具函数放在主包目录下，所有分包都可以引用
usingComponents 引用分包可以引用主包中的自定义组件
独立分包限制独立分包不能依赖主包组件，需要把组件复制到分包内或使用插件

示例：分包引用主包组件

json
{
    "usingComponents": {
        "my-component": "/components/my-component/my-component"
    }
}

三、进阶优化篇 ​

问题 7：什么是分包预下载？如何配置？ ​

解析：

分包预下载是在用户可能进入分包前，提前下载分包代码，减少跳转时的等待时间。

配置方式（在 app.json 中）：

json
{
    "preloadRule": {
        "pages/index/index": {
            "network": "all",
            "packages": ["packageA"]
        },
        "packageA/pages/cat/cat": {
            "network": "wifi",
            "packages": ["packageB"]
        }
    }
}

配置说明：

  • network：all（所有网络）或 wifi（仅WiFi下预下载）
  • packages：要预下载的分包名（name）或 root

最佳实践：

  • 在 TabBar 页面预下载常用分包
  • 在分包入口页预下载下一级分包
  • 注意控制预下载总量，避免流量消耗过大

问题 8：如何查看和分析当前项目的分包体积分布？ ​

解析：

方法一：微信开发者工具

  1. 点击「详情」→「基本信息」→「本地代码」
  2. 查看主包和各个分包的体积
  3. 点击「依赖分析」可以看到具体文件的体积占比

方法二：代码体积分析 开发者工具提供「代码依赖分析」功能，可以：

  • 查看每个包的总大小
  • 分析哪些文件/组件占用了最多空间
  • 发现冗余代码和重复依赖

优化建议：

  • 定期检查分包体积，避免单个分包超过 2MB
  • 使用 webpack-bundle-analyzer 等工具分析依赖
  • 移除未使用的组件和代码

问题 9：独立分包有哪些特殊限制和使用场景？ ​

解析：

独立分包的限制：

限制项说明
不能依赖主包无法使用主包的组件、JS 文件、app.wxss 样式
getApp() 不可用无法获取全局 app 实例
需要自定义 App独立分包需要有自己的 app.js 逻辑
部分 API 受限如 wx.login 等需要在主包中调用

适用场景：

  • 广告落地页（快速打开，无需加载主包）
  • 临时活动页面（用完即走）
  • 分享入口页面（用户可能直接通过分享链接进入）

配置示例：

json
{
    "root": "activity",
    "pages": ["pages/lottery/lottery"],
    "independent": true
}

四、踩坑与最佳实践篇 ​

问题 10：分包开发中常见的坑有哪些？如何避免？ ​

解析：

常见问题原因解决方案
TabBar 页面报错TabBar 页面放在了分包TabBar 页面必须在主包 pages 中
组件找不到分包引用了主包不存在的组件确保公共组件在主包目录下
独立分包白屏独立分包依赖了主包资源检查独立分包的 usingComponents 和 JS 依赖
预下载不生效packages 写错或网络条件不满足检查分包 name 是否正确，network 配置是否合理
路径跳转失败路径写错或分包未配置确保 app.json 中已声明分包，路径以 / 开头
