## 要求

- 若问题和代码相关，回答中给出关键、50 行以内的简短代码示例。
- 回答时，选择合适的逻辑关系组织回答，如：用户视角交互先后顺序、程序运行视角执行先后顺序、开发视角依赖先后顺序、黄金思维圈 why/how/what 顺序、PDCA 法则、STAR 法则等

WebRTC 是一项让浏览器无需插件即可进行点对点实时音视频与数据传输的开源技术，核心价值在于"免插件 + 低延迟 + 端到端加密"。下面按黄金思维圈 Why → How → What 的顺序，逐个回答你的问题。

一、Why：WebRTC 为什么会诞生？ ​

在 WebRTC 出现之前，浏览器原生并不具备人与人之间直接传输音视频数据的能力。要做实时通话，只能依赖 Flash、ActiveX 或各类专有插件，这套方案存在三个难以根治的痛点：

  1. 插件安全隐患：以 Adobe Flash 为例，长期背负安全漏洞负面评价，乔布斯曾公开发信说明 iOS 拒绝 Flash 的原因，Adobe 最终于 2020 年终止服务。浏览器厂商无法控制第三方插件的更新节奏，风险持续放大。
  2. 体验断层：用户必须先下载、安装、更新插件才能通话，"需要准备"才能通信，与 Web 即开即用的理念相悖。
  3. 协议割裂：各家音视频方案私有、专利费用高昂，跨平台、跨浏览器互通几乎不可能。 Google 在 2010 年以 6820 万美元收购 GIPS（Global IP Solutions），获得成熟的音视频编解码引擎；2011 年 5 月将其开源并与 W3C/IETF 推动标准化；2017 年 Safari/Edge 跟进；2021 年 1 月 W3C 与 IETF 同时发布 WebRTC 1.0 为正式标准，标志实时通信真正成为 Web 平台的原生能力。 一句话总结诞生动机：把"高质量、免费、跨平台、免插件"的实时通信能力变成浏览器的标准内置功能。

二、How：WebRTC 的工作原理与开发示例 ​

2.1 整体架构：三大核心 API ​

WebRTC 的能力由三个 JavaScript API 承载，理解它们就理解了 WebRTC 的底层实现：

API作用类比
MediaStream（getUserMedia）采集摄像头/麦克风/屏幕的音视频流"拿原料"
RTCPeerConnection建立 P2P 连接，负责编解码、NAT 穿透、加密、传输"建管道"
RTCDataChannel在 P2P 连接上传输任意二进制/文本数据"传货物"

2.2 连接建立的协议栈 ​

WebRTC 并非单一协议，而是一整套协议组合：

  • 信令：交换会话控制信息，WebRTC 标准未规定信令协议，通常用 WebSocket 自行实现。
  • SDP（Session Description Protocol）：描述媒体能力（编解码、分辨率、带宽），采用 Offer/Answer 模型协商。
  • ICE（Interactive Connectivity Establishment）：候选地址收集与连通性检查框架，整合 STUN 与 TURN。
  • STUN：让 NAT 后的客户端发现自己的公网反射地址（srflx candidate）。
  • TURN：P2P 打洞失败时的中继兜底（relay candidate），稳定但贵。
  • DTLS-SRTP：强制端到端加密，所有媒体流必须加密传输。

2.3 P2P 建连时序 ​

下图展示了从信令握手到媒体通道建立的完整执行时序，这也是 WebRTC 程序运行视角的核心流程：

协商的本质可以概括为两句话：SDP 解决"用什么参数通话"，ICE 解决"怎么连得上"。ICE 候选地址有三类：host（内网地址）、srflx（STUN 反射公网地址）、relay（TURN 中继地址），ICE 框架按优先级尝试所有组合，选出最佳通路。

2.4 内网应用开发示例 ​

下面给出一个可在同一局域网内直接跑通的最小示例。内网环境下 host candidate 可直接连通，因此无需配置 STUN/TURN。核心是三步：采集媒体 → 建立 PeerConnection → 通过信令交换 SDP/ICE。 信令服务器 signaling.js（Node.js，约 20 行）：

js
const { createServer } = require("http");
const { Server } = require("socket.io");
const io = new Server(createServer(), { cors: { origin: "*" } });
io.on("connection", (socket) => {
  socket.on("join", (room) => {
    socket.join(room);
    socket.to(room).emit("peer");
  });
  // 中转 offer / answer / candidate
  ["offer", "answer", "candidate"].forEach((t) =>
    socket.on(t, (data) => socket.to(data.room).emit(t, data)),
  );
});
io.listen(3000, () => console.log("signaling on :3000"));

前端 index.html（关键逻辑，约 45 行）：

html
<video id="local" autoplay muted playsinline></video>
<video id="remote" autoplay playsinline></video>
<script src="/socket.io/socket.io.js"></script>
<script>
  const socket = io("http://localhost:3000");
  const pc = new RTCPeerConnection(); // 内网可不配 iceServers
  const localV = document.querySelector("#local");
  const remoteV = document.querySelector("#remote");
  navigator.mediaDevices.getUserMedia({ video: true, audio: true }).then((stream) => {
    localV.srcObject = stream;
    stream.getTracks().forEach((t) => pc.addTrack(t, stream));
  });
  pc.ontrack = (e) => (remoteV.srcObject = e.streams[0]);
  pc.onicecandidate = (e) =>
    e.candidate && socket.emit("candidate", { room: "r1", candidate: e.candidate });
  socket.emit("join", "r1"); // 进入房间
  socket.on("peer", async () => {
    // 作为 caller 发起呼叫
    const offer = await pc.createOffer();
    await pc.setLocalDescription(offer);
    socket.emit("offer", { room: "r1", offer });
  });
  socket.on("offer", async ({ offer }) => {
    // callee 应答
    await pc.setRemoteDescription(offer);
    const answer = await pc.createAnswer();
    await pc.setLocalDescription(answer);
    socket.emit("answer", { room: "r1", answer });
  });
  socket.on("answer", ({ answer }) => pc.setRemoteDescription(answer));
  socket.on("candidate", ({ candidate }) => pc.addIceCandidate(candidate));
</script>

运行步骤：npm i socket.io → node signaling.js → 两个浏览器标签页打开 index.html，各自会看到本地画面并自动建立连接，对方画面出现在 #remote 中。注意浏览器要求 https 或 localhost 才能授权摄像头。 ​

三、What：WebRTC 的应用场景与延伸问题 ​

3.1 应用场景 ​

WebRTC 的延迟通常可控制在 500ms 以内，远低于传统 CDN 直播（3~10s），这决定了它的应用边界集中在"需要互动"的场景：

场景类别典型应用核心诉求
音视频会议腾讯会议、飞书、Google Meet、Zoom 网页版多人实时、屏幕共享、网络自适应
在线教育VIPKID、双师课堂、互动白板低延迟互动、课堂录制回放
社交娱乐视频聊天、语音房、在线 K 歌、直播连麦高音质、回声消除、连麦无缝
远程医疗/客服远程会诊、视频客服、远程协助高清稳定、端到端加密、合规
IoT 与监控智能摄像头、远程设备控制实时推流、双向音频
实时游戏/云桌面多人游戏语音、云电脑超低延迟数据通道
数据协作文件 P2P 传输、协同编辑、实时字幕RTCDataChannel 低延迟传输
多人会议的拓扑选择是工程上的关键决策，常见的三种架构对比：

3.2 值得探索的延伸问题 ​

围绕 WebRTC 还有不少值得深入的方向，按"媒体/网络/服务端/生态"四个维度梳理：

  1. TURN 服务器何时必须部署？ 对称 NAT、企业严格防火墙场景下 P2P 打洞必然失败，必须用 coturn 等中继兜底，如何评估带宽成本与连通率平衡？
  2. SFU 媒体服务器如何选型？ mediasoup、Janus、LiveKit、Jitsi Videobridge 各自架构与性能差异，哪种适合你的并发模型？
  3. Simulcast 与 SVC 如何优化多人会议？ 多层编码 + 服务端按订阅者带宽选择性转发，是降带宽的关键技术。
  4. WebRTC 在弱网下如何抗丢包？ GCC 拥塞控制、NACK 重传、FEC 前向纠错、RED 音频冗余的协同机制。
  5. WebTransport / WHIP / WHEP 会取代 WebRTC 吗？ 基于 HTTP/3 的新一代低延迟传输，与 WebRTC 的互操作与定位差异。
  6. 端到端加密（E2EE）如何实现？ Insertable Streams（可插入流）API 让应用层在媒体流中注入自己的加密帧，服务端 SFU 不可解密。
  7. WebRTC 与 AI 结合：实时语音识别、增量机器翻译、神经 TTS 流水线如何利用 DataChannel 实现亚秒级延迟翻译。 这些方向覆盖了从底层网络穿透、服务端架构、传输优化到上层 AI 应用的完整链路，可按需深入。
