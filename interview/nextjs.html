引导学习 Next.js 的关键问题（由浅入深） ​

问题列表 ​

  1. Next.js 是什么？它和 React 有什么本质区别？
  2. Next.js 中的文件系统路由是如何工作的？如何创建动态路由？

  4. getServerSideProps、getStaticProps 和 getStaticPaths 分别解决什么问题？
  5. 在 App Router（Next.js 13+）中，服务端组件和客户端组件有什么区别？如何决定使用哪一个？
  6. Next.js 中如何获取数据？举例说明在 Server Component 和 Client Component 中的不同写法。
  7. Next.js 的 API Routes 是什么？如何用它实现简单的后端逻辑？
  8. Next.js 中如何优化图片和字体？next/image 和 next/font 提供了哪些能力？

逐个解析回答 ​

1. Next.js 是什么？它和 React 有什么本质区别？ ​

解析：

  • React 是一个用于构建用户界面的 JavaScript 库，它主要负责视图层（V层）。React 应用默认是客户端渲染（CSR），也就是说，浏览器下载一个几乎空的 HTML 和 JS 文件，然后由 JS 生成页面内容。这会导致首屏加载慢、SEO 不友好。
  • Next.js 是一个基于 React 的全栈框架，它内置了服务端渲染（SSR）、静态站点生成（SSG）、路由、API 路由、优化等能力。它解决了 React 在生产环境中的常见痛点：SEO、首屏性能、路由配置、打包优化等。

本质区别：
React 是“构建 UI 的库”，你需要自己搭配路由、打包工具（Vite/Webpack）、数据请求方案等。Next.js 是“生产级框架”，提供了开箱即用的约定和优化，并且默认在服务器端做更多事情（如预渲染、API 转发）。

2. Next.js 中的文件系统路由是如何工作的？如何创建动态路由？ ​

解析：
Next.js 使用 pages 目录（Pages Router）或 app 目录（App Router）下的文件结构自动定义路由。

Pages Router 示例：

  • pages/index.js → 路由 /
  • pages/about.js → 路由 /about
  • pages/blog/[slug].js → 匹配 /blog/hello-world，slug 参数可通过 useRouter() 或 context.params 获取。
  • 捕获所有路由：pages/docs/[...slug].js → 匹配 /docs/a/b/c。

App Router（推荐新版）：

  • app/page.js → /
  • app/about/page.js → /about
  • app/blog/[slug]/page.js → /blog/hello
  • 使用文件夹 [slug] 表示动态段，通过 params 参数在 page.js 中获取。

动态路由创建关键点：

  • 用方括号 [param] 包裹文件名或文件夹名。
  • 在页面组件中通过 router.query（Pages）或 params（App 中的 page 函数参数）获取值。

3. Next.js 中预渲染（Pre-rendering）的两种形式是什么？它们各自适合什么场景？ ​

解析：
Next.js 默认会对所有页面进行预渲染（除非使用客户端渲染），预渲染分为两种：

形式过程特点适合场景
静态生成（SSG）在构建时生成 HTML速度快、可 CDN 缓存、SEO 友好，但内容不会实时更新博客、文档、产品目录、营销页（数据变化不频繁）
服务端渲染（SSR）每次请求时在服务器生成 HTML内容实时，但响应稍慢、服务器开销大需要实时数据的面板、个性化内容（如用户主页）、社交动态

另外还有增量静态再生（ISR）：SSG 的一种增强，允许在运行时定期重新生成静态页面（revalidate 选项）。

选择原则：
能静态生成就静态生成（提升性能），需要实时数据时才用 SSR。

4. getServerSideProps、getStaticProps 和 getStaticPaths 分别解决什么问题？ ​

解析（针对 Pages Router，App Router 有不同机制但思想类似）：

  • getStaticProps (SSG)：
    在构建时获取数据并生成静态 HTML。

    js
    export async function getStaticProps() {
    const data = await fetch('...');
    return { props: { data } };
}

    解决问题：如何让页面在构建时注入外部数据，同时保持静态特性。

  • getStaticPaths (配合 SSG 用于动态路由)：
    指定哪些动态路径需要在构建时预渲染。

    js
    export async function getStaticPaths() {
    const paths = [{ params: { id: '1' } }, { params: { id: '2' } }];
    return { paths, fallback: false };
}

    解决问题：动态路由（如 /posts/1）无法直接静态生成——需要告诉 Next.js 要生成哪些具体的路径。

  • getServerSideProps (SSR)：
    每次请求时运行，获取实时数据并传入页面组件。
    解决问题：需要每次请求都拿到最新数据的场景（例如用户订单详情、实时仪表盘）。

注意：在 App Router 中，这些函数被取代了——使用 Server Component 的 async 组件直接 fetch 数据，并通过 export const revalidate 或 export const dynamic 控制缓存行为。但理解这些旧 API 依然有助于掌握核心概念。

5. 在 App Router（Next.js 13+）中，服务端组件和客户端组件有什么区别？如何决定使用哪一个？ ​

解析：
App Router 引入了 React Server Components（RSC）。所有组件默认是服务端组件，你可以通过文件顶部的 'use client' 指令显式声明为客户端组件。

特性服务端组件客户端组件
渲染位置服务器（生成 HTML 或 RSC Payload）浏览器（客户端）
能否使用 useState、useEffect否是
能否访问浏览器 API（localStorage, window）否是
数据获取可以直接 async/await + fetch需要使用 useEffect 或 SWR、React Query
打包大小不发送 JavaScript 到客户端（更小）发送 JS，增加 bundle 体积

决定原则：

  • 优先使用服务端组件：除非你需要交互（事件监听）、状态、浏览器 API 或依赖客户端库。
  • 常见的客户端组件场景：按钮、表单、侧边栏切换、画图、使用 useRouter 进行路由跳转（尽管 useRouter 在 App Router 中也可以用在服务端组件？不，next/navigation 的 useRouter 是客户端 hook，只能在客户端组件使用）。
  • 最佳实践：将交互部分抽离成小的客户端组件，外层仍用服务端组件。

6. Next.js 中如何获取数据？举例说明在 Server Component 和 Client Component 中的不同写法。 ​

解析：

  • 服务端组件（App Router 默认）：
    直接使用 async 函数组件 + fetch。Next.js 扩展了原生 fetch，支持缓存和重新验证。

    js
    // app/users/page.js
export default async function UsersPage() {
    const res = await fetch('https://api.example.com/users', { cache: 'force-cache' });
    const users = await res.json();
    return (
        <div>
            {users.map((u) => (
                <p key={u.id}>{u.name}</p>
            ))}
        </div>
    );
}

  • 客户端组件：
    传统 React 方式，使用 useEffect + fetch，或者用 SWR、React Query 等库。

    js
    'use client';
import { useEffect, useState } from 'react';

export default function ClientUsers() {
  const [users, setUsers] = useState([]);
  useEffect(() => {
    fetch('/api/users')
      .then(res => res.json())
      .then(setUsers);
  }, []);
  return <div>{users.map(u => ...)}</div>;
}

  • 混合模式：
    也可以在服务端组件获取数据后，通过 props 传递给客户端组件（避免重复请求）。

提示：在服务端组件中 fetch 默认会自动去重并缓存（类似 cache: 'force-cache'），要获取实时数据可设置 cache: 'no-store' 或使用 next: { revalidate: 60 }。

7. Next.js 的 API Routes 是什么？如何用它实现简单的后端逻辑？ ​

解析：
API Routes 允许你在 Next.js 项目中创建后端接口，无需额外启动一个 Node 服务器。

  • Pages Router：pages/api/*.js 文件导出一个函数，接收 req 和 res。
  • App Router：app/api/route.js 导出 HTTP 方法对应的函数（GET, POST 等）。

示例（App Router）：

js
// app/api/hello/route.js
export async function GET() {
    return Response.json({ message: 'Hello Next.js' });
}

export async function POST(request) {
    const body = await request.json();
    // 处理数据...
    return Response.json({ received: body });
}

前端可以调用 /api/hello 接口。

用途：

  • 代理外部 API（隐藏密钥）
  • 处理表单提交、数据库操作（配合 Prisma、MongoDB）
  • 认证（JWT、NextAuth.js 后台）
  • 文件上传（配合 multer 等，但注意 Next.js 内置 bodyParser 限制）

注意：API Routes 不是完全的独立后端服务（无中间件、无 WebSocket 等），但对于小规模全栈项目足够。

8. Next.js 中如何优化图片和字体？next/image 和 next/font 提供了哪些能力？ ​

解析：

next/image：

  • 自动优化图片（延迟加载、WebP/AVIF 格式、尺寸适配）。
  • 防止 CLS（布局偏移）—— 要求设置 width 和 height 或 fill 属性。
  • 内置懒加载，仅在进入视口时加载。
  • 示例：
    js
    import Image from 'next/image';
<Image src="/hero.jpg" width={800} height={400} alt="Hero" />;

next/font：

  • 自动优化字体（自托管 Google Fonts 等，避免外部请求）。
  • 支持 CSS size-adjust 和 fallback，减少布局偏移。
  • 默认开启 display: swap 并内联字体 CSS。
  • 示例：
    js
    import { Inter } from 'next/font/google';
const inter = Inter({ subsets: ['latin'] });
<html className={inter.className}>...

两者都是 Next.js 优化核心性能的关键工具，无需手动处理就能显著提升 LCP（Largest Contentful Paint）和整体体验。

9. 什么是 Next.js 中的 Middleware？它可以用来做哪些事情？ ​

解析：
Middleware 是一种在请求到达最终路由之前执行的代码，运行在 Edge Runtime（轻量、快速）。你需要在项目根目录创建 middleware.js（或 middleware.ts）。

基本结构：

js
// middleware.js
import { NextResponse } from 'next/server';

export function middleware(request) {
    const token = request.cookies.get('token');
    if (!token && request.nextUrl.pathname.startsWith('/dashboard')) {
        return NextResponse.redirect(new URL('/login', request.url));
    }
    return NextResponse.next();
}

// 配置匹配路径
export const config = { matcher: ['/dashboard/:path*'] };

典型用途：

  • 身份验证与授权（重定向未登录用户）
  • 请求改写（如 A/B 测试、重写 URL 到不同的页面）
  • 地理定位（根据请求头判断国家/城市，返回不同内容）
  • Bot 检测（识别搜索引擎爬虫，返回特殊页面）
  • 日志记录（记录请求路径、耗时）

Middleware 不能直接操作 DOM 或发送 HTML，但它非常适合做路由级别的控制和预处理。

10. 从开发到部署，如何对一个 Next.js 项目进行性能分析和常用优化？ ​

解析：

开发阶段：

  • 使用 Next.js 内置的 Bundle Analyzer（@next/bundle-analyzer）查看打包体积。
  • 启用 React Strict Mode。
  • 使用 next dev --turbo（实验性）加快本地编译。

构建与部署前优化：

  1. 图片：全面使用 next/image。
  2. 字体：使用 next/font，避免外部字体链接。
  3. 脚本：使用 next/script 控制第三方脚本加载时机（strategy="lazyOnload"）。
  4. 动态导入：对大型组件使用 next/dynamic 懒加载客户端组件。
  5. 代码分割：Next.js 自动按路由分割，但注意避免在服务端组件中引入巨大依赖。
  6. 缓存策略：
    • 静态资源（/_next/static）默认长期缓存。
    • 对于 SSG 页面可设置 CDN 缓存。
    • 使用 ISR 或 fetch 的 revalidate 平衡新鲜度。

运行时监控：

  • 使用 Vercel Analytics（或自行集成 Core Web Vitals 报告）。
  • 检查 LCP、FID、CLS 指标。
  • 利用 next dev --experimental-https 模拟生产环境。

部署建议：

  • 推荐部署到 Vercel（零配置优化、自动处理 CDN、边缘函数）。
  • 自托管时使用 output: 'standalone' 减少镜像体积。
  • 配置 remotePatterns 限制外部图片域名。

常用命令行：

bash
npm run build   # 构建后查看各页面大小和第一层 JS 体积
npx next build --analyze   # 可视化分析

通过系统性的分析和优化，Next.js 项目通常可以达到接近完美的性能分数。
